## 一、设计目标与原则

### 1.1 目标

在**网络安全漏洞管理平台**（以下简称「本平台」）与**外部第三方系统**之间，提供可复用、可扩展、可治理的集成能力，使各类合作方（SOC、SIEM/SOAR、ITSM、资产平台、监管报送代理、定制开发系统等）通过统一契约接入，而无需为每个系统单独编写点对点接口。

| 能力域 | 说明 |
|--------|------|
| 任务编排 | 创建/查询扫描与处置类任务 |
| 漏洞数据 | 实例级漏洞的查询、订阅、批量导出 |
| 生命周期 | 验证、**修复/备案并列处置**、核验等状态推进 |
| 事件通知 | 任务完成、状态变更、数据就绪等异步回调 |
| 合规导出 | 按监管/行业配置生成上报结构（如部侧 `vulInfoLst`） |

### 1.2 设计原则

| 原则 | 含义 |
|------|------|
| **内外分离** | 对外暴露「开放平台 API + 标准事件」；对内对接 SVMP 等执行引擎 |
| **领域统一** | 平台维护一套**规范域模型**（状态机、标识、码表）；第三方通过映射配置对齐 |
| **适配可插拔** | 协议（REST/Webhook/Kafka）、字段、鉴权按**接入方（Partner）**配置，不写死在代码分支 |
| **合规可配置** | 工信部 2025 等要求作为**合规配置档（Compliance Profile）**，而非唯一 API 形态 |
| **幂等可观测** | 全链路 `externalRefId`、请求日志、对接指标、失败重试与死信 |

### 1.3 与 V1.1 的关系

- V1.1 中的 SOC 专用路径（`/api/soc/*`）、`socTaskId` 等，在 V2.0 中收敛为**开放平台 API** + **SOC 接入配置档（Partner Profile）**。
- 部侧字段（`vulInfoID`、`vulInfoStat`、`lvRsn` 等）为平台**规范域模型**主标识；开放平台 API **沿用同名**，与表34 / 报送结构一致。

## 二、总体架构

### 2.1 逻辑分层

```text
┌─────────────────────────────────────────────────────────────────┐
│                    外部第三方系统（Partner）                      │
│   SOC / SIEM / ITSM / 资产平台 / 监管代理 / 自研系统 …           │
└───────────────┬───────────────────────────────┬─────────────────┘
                │ REST / Webhook                  │ Kafka（可选）
                ▼                                 ▼
┌─────────────────────────────────────────────────────────────────┐
│              集成网关 Integration Gateway                          │
│  · 鉴权与限流  · 幂等  · 开放平台 API  · 回调投递  · 审计         │
└───────────────┬─────────────────────────────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────────────────────────────┐
│              集成域服务 Integration Domain                       │
│  · Partner 注册  · 工单编排器（接口3→4）  · vulInfoRange 解析器   │
│  · 规范状态机  · 事件总线  · MIIT-2025 Profile（入站+回传）        │
└───────────────┬───────────────────────────────┬─────────────────┘
                │                                 │
                ▼                                 ▼
┌───────────────────────────┐     ┌───────────────────────────────┐
│   执行引擎适配器            │     │   部侧合规适配器（Ingress/Egress）│
│   SVMP Connector（v2.8）   │     │   接口3 入站解密 · 接口4 回传加密 │
│   （可扩展其他扫描源）       │     │   vulInfoLst 组装 · 周期进度上报  │
└───────────────────────────┘     └───────────────────────────────┘
```

### 2.2 核心概念

| 概念 | 说明 |
|------|------|
| **Partner（接入方）** | 已注册的第三方系统，含鉴权、回调、能力集、字段映射 |
| **External Ref（外部引用）** | 第三方幂等键，如 `extTaskId`、`extVulnId`；平台侧映射内部 ID |
| **Canonical Model（规范模型）** | 平台统一的漏洞实例、任务、状态、事件定义 |
| **Capability（能力项）** | Partner 可启用的 API/事件子集，如仅查询、或含写回 |
| **Compliance Profile（合规档）** | 如 `MIIT-2025`，定义导出字段、码表、工单类型映射 |
| **Connector（连接器）** | 对接 SVMP 等后端系统的适配实现 |

### 2.3 数据流（典型）

**路径 A · 部侧工单（必遵接口3 说明）**

```text
部侧下发接口3（加密）→ Ingress 解密 → 解析 orderSubType / vulInfoRange
  → WorkOrder Orchestrator → SVMP 任务/处置
  → 实例状态推进（§7.3.3 场景表）→ 接口4 回传 vulInfoLst（加密）
```

**路径 B · 第三方开放平台**

```text
Partner 创建任务（extTaskId）→ Gateway 鉴权 → SVMP Connector
  → TASK_COMPLETED 事件 → Partner Webhook
Partner verify/remediate → 状态机 → 若关联 orderID 则纳入同工单回传队列
```

## 三、规范域模型（漏洞实例 · 部侧字段标识）

> **概念**：资产上的具体漏洞记为**系统漏洞 / 漏洞实例**（部侧 `vulInfoLst` · `InstVulLst`）。  
> **标识**：开放平台 API **沿用部侧字段名**（表34 等），与报送结构一致，不做二次改名。

### 3.1 字段命名原则

| 类别 | 规则 |
|------|------|
| 部侧已定义（表34 `InstVulLst`、修复台账、A.8/A.23 等） | API **同名**，类型与码表与部侧一致 |
| 引擎 / Partner 扩展 | 见 §3.2，不与部侧字段混用 |

### 3.3 修复台账 `remedInfo`（部侧）

| 字段 | 说明 |
|------|------|
| `srcMethod` | 处置方式码（1050/1051/1052 等） |
| `remedDesc` | 修复说明 |
| `fixLnk` | 补丁链接（`srcMethod=1050` 时必填） |
| `defDev` | 防护设备（1051/1052 时必填） |
| `remedTime` | 修复耗时 |

### 3.4 漏洞实例状态 `vulInfoStat`（部侧附录 A.8）

| vulInfoStat | 说明 | 阶段 |
|-------------|------|------|
| 0 | 潜在预警 | 预警 |
| 1 | 初始发现 | 识别 |
| 2 | 已验证有效 | 识别 |
| 3 | 已验证误报 | 修复（终态阻断） |
| 5 | 已修复 | 修复 |
| 6 | 核验修复 | 修复 |
| 7 | 核验未修复 | 识别 |
| 8 | 验证失败 | 识别 |
| 9 | 修复失败 | 识别 |
| 10 | 核验失败 | 识别 |

**状态机守卫**（全局，所有 Partner 共用；数值即 `vulInfoStat`）：

```text
vulInfoStat=1 ──verify──▶ 2 | 3
vulInfoStat=3 ──▶ 禁止进入处置阶段（修复/备案）

处置阶段（修复与备案并列，二选一）：
  vulInfoStat ∈ {2, 7}
    ├─ remediate（可修复）  ──▶ 5，清空 lvRsn
    └─ archive（不可修复备案）──▶ 9，必填 lvRsn + 备案扩展

vulInfoStat=5 ──引擎核验──▶ 6 | 7
```

### 3.5 未修复原因 `lvRsn`（部侧附录 A.23）

| lvRsn | 说明 |
|-------|------|
| 101–109, 999 | 见 §3.6 填写场景 |

### 3.6 处置阶段：修复与备案并列

验证有效（`vulInfoStat=2`）后进入**处置阶段**。该阶段包含两条**互斥**分支，与部侧修复工单（orderSubType **33/43**）中目标态 **5 / 9** 对应：

| 分支 | 业务含义 | 平台 API | 终态 `vulInfoStat` | `lvRsn` |
|------|----------|----------|-------------------|---------|
| **修复** | 可修复，已完成修复动作 | `POST .../remediate` | **5** | 清空 |
| **备案** | 不可修复，按规定备案 | `POST .../archive` | **9** | **必填**（101–109 / 999） |

**规则**：

1. 同一实例在处置阶段**只能择一**：已走 `remediate` 后不可再 `archive`，反之亦然（除非运营配置允许纠错流程，默认禁止）。
2. **可修复** → 必须走修复分支，终态 **5**；部侧回传须含 `remedTime`、`remedDesc`、`procMethod`（如 1050）等修复台账字段。
3. **不可修复** → 必须走备案分支，终态 **9（修复失败）**；部侧回传须含 **`lvRsn`** 及备案说明字段 **`archiveReason`**、`provincialFields`（省侧待扩充，见 §5.8）。
4. 部侧规范中 `lvRsn` 通常用于未修复态（1/2/7）；本方案在备案分支采用 **态 9 + lvRsn** 表达「修复工单语境下的不可修复」，与接口示例 V2.2 修复工单目标态位图（含 **9**）一致；若省侧报送校验有差异，以 `MIIT-2025` Profile 配置覆盖。

**备案补充字段（企业内部，非部侧状态码）**：

| 字段 | 必填 | 说明 |
|------|:---:|------|
| `archiveReason` | ✓ | 备案说明 |
| `operator` | ✓ | 操作人 |
| `approvedBy` | ○ | 审批人 |
| `recordAt` | ○ | 备案时间 |
| `provincialFields` | ○ | 省侧扩展 JSON，待扩充 |

### 3.7 生命周期阶段（与部侧工单对照）

> 工单类型以 **`orderID` + `orderType` + `orderSubType`** 三元组区分（接口示例 V2.2 · 接口3 说明）。

| 平台阶段 | 部侧请求 orderSubType | 部侧回传 orderSubType | 开放平台能力 / 内部编排 |
|----------|----------------------|----------------------|-------------------------|
| 排查 | **31** | **41** | `tasks.create`；目标态 → **1** 初始发现 |
| 验证 | **32** | **42** | `instances.verify`；目标态 → **2** / **3** / **8** |
| 处置·修复 | **33** | **43** | `instances.remediate` → **5** |
| 处置·备案 | **33** | **43** | `instances.archive` → **9** + `lvRsn`（与修复共用修复工单，按终态区分回传） |
| 核验 | **34** | **44** | 引擎 `verify` + 回传；目标态 → **6** / **7** / **10** |
| 排查处置（通用） | 3 | 4 | 与排查共用编排，范围表达式不同 |

**部侧下发工单参数（`vulInfoTktReqParams`）与平台映射**：

| 部侧字段 | 平台处理 |
|----------|----------|
| `vulInfoRange` | 解析为实例筛选条件（见 §7.3.2） |
| `pwDictRange` | 口令类工单；与产品漏洞范围二选一 |
| `dstVulInfoStat[]` | 工单目标状态位图（`-1` 表示该位无效） |
| `procMethod` | 下发可为类别码（如 108、102）；执行后回传明细码（如 1020、1050） |
| `srcVulInfoStat` | **部侧不下发**；源态筛选由 `vulInfoRange` 中 `vulInfoStat` 表达 |
| `vptModID` | 修复工单（33）常指定；映射引擎模板/模型 |
| `procTime` / `timePerd` | 处置时限与周期上报进度 |
| `tktVulNum` / `tktAstNum` | 工单漏洞数/资产数约束 |

## 五、开放平台 API（对外统一契约）

> **Base Path**：`/api/open/v1`  
> **OpenAPI 3.1**：[`openapi/v1/openapi.yaml`](./openapi/v1/openapi.yaml)（可导入 Swagger UI / Postman / codegen）  
> **鉴权**：`Authorization: Bearer <accessToken>` 或 `X-Api-Key` + `X-Signature`（HMAC-SHA256）  
> **Partner 识别**：Token 绑定 `partnerId`，路径中不再重复传 partner（管理接口除外）

### 5.1 通用约定

| 项 | 约定 |
|----|------|
| 幂等 | **仅创建任务**必填 `extTaskId`（或 `Idempotency-Key`）；查询/处置类接口不要求 |
| 任务查询 | 调用 SVMP 时**仅传引擎 `taskId`**，不传 `extTaskId`（见 §6.3） |
| 分页 | `page`（从 1）、`size`（≤1000） |
| 时间 | ISO 8601 UTC，`2026-05-17T08:00:00Z` |
| 响应包装 | `{ "code": 0, "message": "ok", "data": { ... }, "requestId": "..." }` |
| 错误码 | 见 §5.10 |

### 5.2 接口文档体例说明

各接口统一采用下列表格（无该项则标注「—」）：

| 表格 | 内容 |
|------|------|
| **路径参数** | URL 中的变量 |
| **查询参数** | Query String |
| **请求头** | 除鉴权外的补充头 |
| **请求体** | JSON Body 字段 |
| **响应 data** | 业务成功时 `data` 对象字段 |
| **状态约束** | 前置/后置条件 |

**字段列定义**：`参数` · `类型` · `必填`（✓ 必填 / ○ 可选 / 条件 条件必填）· `说明`

**通用请求头**（写接口建议携带）：

| 参数 | 类型 | 必填 | 说明 |
|------|------|:---:|------|
| Authorization | string | ✓ | `Bearer <accessToken>`，或改用 `X-Api-Key` + `X-Signature` |
| Idempotency-Key | string | ○ | 写操作幂等键 |
| Content-Type | string | ✓ | `application/json` |

**通用响应包装**（所有接口 HTTP 200 时 body 结构）：

| 参数 | 类型 | 必填 | 说明 |
|------|------|:---:|------|
| code | int | ✓ | 业务码，`0` 为成功，见 §5.9 |
| message | string | ✓ | 描述信息 |
| data | object | ○ | 成功时业务数据；失败时常为 `null` |
| requestId | string | ✓ | 请求追踪 ID |

### 6.1 适配器类型

| 适配器 | 方向 | 用途 |
|--------|------|------|
| **Open API Adapter** | 入站 | 第三方 → 平台（主通道） |
| **Webhook Adapter** | 出站 | 平台 → 第三方回调 |
| **Kafka Adapter** | 双向可选 | 高吞吐、多订阅 |
| **Batch Export Adapter** | 出站 | SFTP/对象存储批量文件 |
| **Regulatory Ingress Adapter** | 入站 | 部侧工单/指令解析（若平台承担报送中枢） |

### 6.2 映射配置（Mapping Profile）示例

```yaml
profileId: partner-soc-default
partnerType: SOC
inbound:
  extTaskId: $.socTaskId          # 兼容 V1.1 字段名
  targets: $.targets
outbound:
  listItem: {}                     # 主字段与部侧同名，无需映射
complianceExport:
  MIIT-2025:
    vulInfoLst: {}                 # InstVulLst 与 API 字段 1:1 组装
```

新增 Partner 时优先**配置映射**，避免新增代码分支；仅当协议差异过大时新增 Adapter 实现。

### 6.3 SVMP 引擎连接器（内置 Connector）

| 开放平台动作 | SVMP 接口 | 下发 SVMP 的关键参数 |
|--------------|----------|----------------------|
| 创建任务 | POST `/task/create`（先 GET `/device/list`） | 扫描目标、模板等；**不含** `extTaskId` |
| 查询进度 | GET `/task/progress` | **`taskId`（SVMP）仅此一项** |
| 实例列表 | POST `/vuln/disposal/list` | **`taskId`（SVMP）** + 筛选条件 |
| 实例详情 | GET `/vuln/disposal/detail` | `vulnDisposalId` 等 |
| 状态处置 | POST `/vuln/disposal/disposal` | 处置参数 + `vulnDisposalId` |
| 修复核验 | POST `/vuln/disposal/verify` | 核验参数 + `vulnDisposalId` |

**任务幂等分工**：

```text
Partner extTaskId  →  仅 POST /tasks（平台幂等、映射表）
平台 taskId        →  对外查询路径 /tasks/{taskId}
SVMP taskId        →  创建成功后写入映射表；后续读请求只带 SVMP taskId
```

| 场景 | Partner 是否传 extTaskId | 调 SVMP 参数 |
|------|-------------------------|--------------|
| 创建任务 | ✓ 必填 | 扫描业务字段；返回 SVMP taskId 落库 |
| 查询进度 | ✗ 不需要 | 仅 SVMP `taskId` |
| 任务列表（平台） | ○ 可选过滤 | 不调用 SVMP 进度接口 |
| 漏洞实例搜索 | ○ 可选；可只传 `taskId` | 解析后仅 SVMP `taskId` |
| 验证/修复/备案 | ✗ 不需要（实例以 vulInfoID 为主） | 与任务键无关 |

| SVMP 字段 | 部侧/API 字段 |
|----------|----------------|
| vulDisposalId | vulnDisposalId |
| cveId | orgVulId |
| assetKey | vulNetAddr |
| port | vulPort |
| service | vulSvc |
| vulName | vulName |
| vulLevel | vulLevel |
| vulSolution | remedDesc |

## 七、部侧合规：工单入站（接口3）与回传（接口4）

> **必遵来源**：《基础电信企业安全漏洞管理平台接口示例 2025-V2.2》· **接口3：系统漏洞排查/验证/修复/核验工单请求** · **说明**章节，及配对接口4回传。  
> 本平台作为省/集团侧漏管平台时，**Regulatory Ingress Adapter** 必须完整实现下列约束；开放平台 API 与 SVMP 编排均不得违背部侧工单语义。

### 7.1 传输与报文信封（L2）

除认证接口（接口1/2 `provisionOrder`）外，部侧业务接口均需 **SM2RSA + SM4-GCM** 混合加密（双向）。

**请求/响应外层（固定参数 L2）**：

| 字段 | 方向 | 说明 |
|------|------|------|
| `orderID` | 请求/响应 | 工单唯一 ID；参与 sign |
| `orgCode` | 请求 | 组织代码 |
| `ispCode` | 请求 | 运营商代码 |
| `ctxCode` | 请求 | 上下文 |
| `reqMsgCnt` | 请求 | **密文**；解密后为工单业务 JSON |
| `statusCode` / `statusText` | 响应 | 接收结果 |
| `rspMsgCnt` | 响应 | **密文**；解密后为回执 JSON |

**请求头（加密）**：`X-Enc-Key`、`X-Enc-Key-G`（集团场景）、`X-Enc-Nonce`、`X-Enc-Auth-Tag`（JSON）、`X-Enc-Auth-Tag-File`（文件接口）。

**sign 范围**：L2 固定参数 — 请求 `orderID, ispCode, orgCode, ctxCode`；响应 `orderID, statusCode, statusText, reqMsgCnt/rspMsgCnt`。

### 7.2 接口3 · reqMsgCnt 明文结构

```json
{
  "orderType": 1,
  "orderSubType": 31,
  "timeStamp": "1691058006",
  "sign": "...",
  "vulInfoRange": "<Base64(查询表达式)>",
  "pwDictRange": "<Base64(可选，口令字典范围)>",
  "procTime": "20230803181859-20230810181859",
  "timePerd": { "unit": 1, "perd": 1 },
  "vulInfoTktReqParams": {
    "vptModID": "...",
    "srcTktRole": 6,
    "dstTktRole": 6,
    "dstVulInfoStat": [-1, -1, -1, -1, 5, -1, -1, -1, 9, -1],
    "procMethod": 1050,
    "tktPriority": 3,
    "tktVulNum": 2,
    "tktAstNum": 1,
    "tktSLA": "30,70",
    "tktInfo": "附加说明"
  },
  "data": {
    "numFiles": 1,
    "fileInfoLst": [{ "name": "...", "dataType": "json", "svcType": 2 }]
  }
}
```

> 示例中 `srcVulInfoStat` **不会出现于部侧下发**（说明原文）；源状态筛选仅通过 `vulInfoRange` 内 `vulInfoStat` 条件表达。

**平台入站处理流水线**：

```text
部侧 POST（加密 L2）
  → 解密 reqMsgCnt → 校验 sign / 幂等 orderID
  → 解析 orderSubType → 工单编排器（WorkOrder Orchestrator）
  → 解析 vulInfoRange / pwDictRange → 实例筛选
  → 映射 procMethod / dstVulInfoStat → 引擎任务或 Partner 事件
  → 异步执行 → 接口4 回传（加密 rspMsgCnt + vulInfoLst）
```

### 7.3 接口3 · 说明章节（设计必遵）

#### 7.3.1 范围表达式 `vulInfoRange` / `pwDictRange`

| 规则 | 要求 |
|------|------|
| 互斥 | **`pwDictRange` 与产品漏洞范围二选一**：`pwDictRange` 有值则按**口令字典**工单处理，不对 `vulKeys` 查询（表达式视为 `vulKeys is null`） |
| 编码 | `vulInfoRange`、`pwDictRange` 为 **Base64** 包裹的查询表达式 |
| 节点连接 | 复合参数叶节点用 **`.`** 连接（如 `vulKeys.vulID`、`assetInfoRange.assetName`） |
| 列表节点 | **不支持**对列表类型节点查询 |
| 连接符 | 统一**小写**（`and` / `or` / `in` / `not in`） |
| 一级节点 | 仅支持 **`and`**，且用**英文括号**分组 |
| 二级节点 | 支持 `and`、`or`、`in`、`not in` 等 |
| 必含节点 | 复合结构中 **`vulKeys`、`assetInfoRange`、`vulInfoStat` 必传**（其余非必传） |
| 一级检索 | 需支持**一级节点检索**能力 |

**`vulInfoStat` 在表达式中的语义**（说明原文）：

| 写法 | 含义 |
|------|------|
| `vulInfoStat in (0,1)` | 查询状态为 0、1 的漏洞 |
| `vulInfoStat = -1` | 该条件**无效**，即不按状态过滤（含历史未发现） |

**表达式示例**（实现解析器时需通过）：

```text
Q1: vulInfoID = 'vid_0001'

Q2: assetInfoRange.assetName like '%linux%' and vulInfoStat != 0

Q3: (vulKeys.vulID = 'MVM-2019-1696145560773468160'
      and (assetInfoRange.assetName like '%linux%' or assetInfoRange.ip = '10.11.10.104'))
     or vulInfoID = 'vid_0001'
```

**平台实现**：Ingress 层内置 **VulInfoRange Parser**（或 ANTLR/规则引擎），输出规范域 `InstanceQuery`；禁止将原始表达式透传 SVMP SQL。

#### 7.3.2 `dstVulInfoStat` 位图数组

部侧以**下标对应状态码位**的数组表达目标态集合，**`-1` 表示该位不参与**。

示例（修复工单）：`[..., 5, ..., 9, ...]` 表示目标包含「已修复(5)」「修复失败(9)」。

平台编排时：

1. 解析非 `-1` 下标 → 集合 `TargetStatuses`  
2. 结合 `orderSubType` 与 §7.3.3 场景表校验合法性  
3. 执行完成后，实例最终 `vulInfoStat` 必须落在 `TargetStatuses` 之一（或按业务只达成其中一个）

#### 7.3.3 四阶段业务场景（说明表 · 以部侧实际下发为准）

| 工单业务 | 源状态（筛选） | 目标状态（dst） | 典型 procMethod（下发/回传） |
|----------|----------------|-----------------|-------------------------------|
| **排查** | 潜在预警(0) / 不指定 | **1** 初始发现 | **1020** 指纹插件 · **1021** POC · **1022** 混合扫描 · **1023** 线下导入 · **1024** 本地发现 · **1027** 登录扫描 · **1028** 连通性 · **1030–1032** 暴破/配置分析 |
| **验证** | **1** 初始发现 | **2** 有效 · **3** 误报 · **8** 验证失败 | **1026** 交叉扫描验证 · **1040** EXP 利用 |
| **处置（修复）** | **2** 已验证有效 | **5** 已修复 | **1050–1053** 等 |
| **处置（备案）** | **2** 已验证有效 | **9** 修复失败 | 无独立 procMethod 类别码时回传按实际备案说明；`lvRsn` 必填 |
| **核验** | **5** 已修复 | **6** 核验修复 · **7** 核验未修复 · **10** 核验失败 | **1030–1032** · **1060** 修复核验自适应扫描 · **1061** 攻击模拟扫描 |

**与开放平台状态机对齐**：

```text
orderSubType 31 → 创建扫描任务 → 实例落库 vulInfoStat=1
orderSubType 32 → 触发 verify 编排 → 2 / 3 / 8
orderSubType 33 → 处置编排：可修复 → remediate → **5**；不可修复 → archive → **9** + lvRsn（vptModID 绑定模板）
orderSubType 34 → 触发 SVMP verify → 6 / 7 / 10
```

**procMethod 编码规则**（说明原文）：

- **下发**可指定台账类别/处置大类（如 `108`、`102`）  
- **回传**须细化到具体处置方式（如 `1020`、`1021`、`1050`）  
- 台账日志 `logType` 与 `srcMethod` 与 `procMethod` 保持一致对应（除 §7.3.4 特殊情形）

#### 7.3.4 处置方式 1060 特殊说明（BAS 规范未下发前）

当工单指定 **1060 修复核验自适应扫描** 时：

- 台账日志格式同「系统漏洞排查、弱口令扫描」  
- `logInfoReqParams.logType = 1060`  
- 实际处置方式可能为 102/103 类，`srcMethod` 表现为 1020 等  
- **其他处置方式**下，台账 `logType` 与系统漏洞 `procMethod` **一致对应**

编排器需单独分支：1060 任务 → 生成排查类日志模板 + 内部映射真实 procMethod。

#### 7.3.5 其他说明项

| 项 | 要求 |
|----|------|
| `timePerd` | 下级按指定周期**上报处置进度**（与接口4回传频次联动） |
| `transID` | 系统漏洞**回传**工单 `vulInfoTktRspParams.transID` 按序号递增（接口4） |
| `vptModID` | **修复工单（33）** 常指定；映射 SVMP `scanTemplateId` / VPT 模型 |
| 附件 `data.fileInfoLst` | 支持 JSON 等附件；解密后落对象存储，关联工单 |

### 7.4 接口4 · 工单回传（概要）

省侧 → 部侧，外层仍为 L2 加密；`reqMsgCnt` 明文核心：

| 块 | 说明 |
|----|------|
| `orderType` / `orderSubType` | 与请求对应，回传为 **41/42/43/44** |
| `vulInfoLst` | 表34 嵌套结构（`comVulLst` → `instVulLst`） |
| `vulInfoTktRspParams` | `transID`、`prcVulNum`、`dstVulInfoStat[]`、`sucVulInfoNum[]`、`exRsnLst` 等 |

**Exporter 职责**：将规范域实例批量转换为 `vulInfoLst`；`vulInfoTktRspParams.dstVulInfoStat` 与实例实际终态数组对齐。

### 7.5 部侧工单 ↔ 开放平台 ↔ SVMP 映射总表

| orderSubType | 平台编排动作 | SVMP / 引擎 | 开放平台（第三方可选） |
|--------------|-------------|-------------|------------------------|
| 31 | `WorkOrder.scan` | `POST /task/create` | `POST /tasks` |
| 32 | `WorkOrder.verify` | `POST /vuln/disposal/disposal` | `POST /instances/{id}/verify` |
| 33 | `WorkOrder.remediate` 或 `WorkOrder.archive` | disposal + `vptModID` | `POST .../remediate` 或 `POST .../archive` |
| 34 | `WorkOrder.verifyFix` | `POST /vuln/disposal/verify` | 事件通知 Partner |
| 41–44 | `WorkOrder.report` | 汇总实例 | `EXPORT_READY` 事件 |

### 7.6 Compliance Profile 与第三方 API 的关系

```text
                    ┌── 接口3 入站（部侧，加密）
                    ▼
            WorkOrder Orchestrator
                    │
     ┌──────────────┼──────────────┐
     ▼              ▼              ▼
 Open API      SVMP Connector   接口4 回传（部侧，加密）
（第三方）                      vulInfoLst
```

- **第三方开放平台 API** 不替代部侧接口3；Partner 触发的任务若需报送部侧，必须能关联到 `orderID` 或进入同一编排上下文。  
- **Compliance Profile `MIIT-2025`** = 接口3/4 加解密 + 表达式解析 + 表34 组装 + 场景表校验。

### 7.7 扩展其他 Profile

未来可增 `CVE-JSON`、`STIX` 等行业格式；**部侧 MIIT-2025 工单为当前强制 Profile**。
