## 一、设计目标与原则

### 1.1 目标

在**网络安全漏洞管理平台**（以下简称「本平台」）与**外部第三方系统**之间，提供可复用、可扩展、可治理的集成能力，使各类合作方（SOC、SIEM/SOAR、ITSM、资产平台、监管报送代理、定制开发系统等）通过统一契约接入，而无需为每个系统单独编写点对点接口。

| 能力域 | 说明 |
|--------|------|
| 任务编排 | 创建/查询扫描与处置类任务 |
| 漏洞数据 | 实例级漏洞的查询、订阅、批量导出 |
| 生命周期 | 验证、修复标记、不可修复备案、核验等状态推进 |
| 事件通知 | 任务完成、状态变更、数据就绪等异步回调 |
| 合规导出 | 按监管/行业配置生成上报结构（如部侧 `vulInfoLst`） |

### 1.2 设计原则

| 原则 | 含义 |
|------|------|
| **内外分离** | 对外暴露「开放平台 API + 标准事件」；对内对接 SVMP 等执行引擎 |
| **领域统一** | 平台维护一套**规范域模型**（状态机、标识、码表）；第三方通过映射配置对齐 |
| **适配可插拔** | 协议（REST/Webhook/Kafka）、字段、鉴权按**接入方（Partner）**配置，不写死在代码分支 |
| **合规可配置** | 工信部 2025 等要求作为**合规配置档（Compliance Profile）**，而非唯一 API 形态 |
| **幂等可观测** | 全链路 `externalRefId`、请求日志、对接指标、失败重试与死信 |

### 1.3 与 V1.1 的关系

- V1.1 中的 SOC 专用路径（`/api/soc/*`）、`socTaskId` 等，在 V2.0 中收敛为**开放平台 API** + **SOC 接入配置档（Partner Profile）**。
- 部侧码表（`vulInfoStat`、`lvRsn`、`orderSubType`）保留为平台**规范域模型**；对外 API 使用语义化枚举，由平台映射到码表。

## 二、总体架构

### 2.1 逻辑分层

```text
┌─────────────────────────────────────────────────────────────────┐
│                    外部第三方系统（Partner）                      │
│   SOC / SIEM / ITSM / 资产平台 / 监管代理 / 自研系统 …           │
└───────────────┬───────────────────────────────┬─────────────────┘
                │ REST / Webhook                  │ Kafka（可选）
                ▼                                 ▼
┌─────────────────────────────────────────────────────────────────┐
│              集成网关 Integration Gateway                          │
│  · 鉴权与限流  · 幂等  · 开放平台 API  · 回调投递  · 审计         │
└───────────────┬─────────────────────────────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────────────────────────────┐
│              集成域服务 Integration Domain                       │
│  · Partner 注册  · 能力协商  · 映射配置  · 外部引用映射表          │
│  · 规范状态机  · 事件总线  · 合规导出（Profile）                   │
└───────────────┬───────────────────────────────┬─────────────────┘
                │                                 │
                ▼                                 ▼
┌───────────────────────────┐     ┌───────────────────────────────┐
│   执行引擎适配器            │     │   数据出口适配器                │
│   SVMP Connector（v2.8）   │     │   部侧报送 / SFTP / 消息队列    │
│   （可扩展其他扫描源）       │     │   （按 Compliance Profile）     │
└───────────────────────────┘     └───────────────────────────────┘
```

### 2.2 核心概念

| 概念 | 说明 |
|------|------|
| **Partner（接入方）** | 已注册的第三方系统，含鉴权、回调、能力集、字段映射 |
| **External Ref（外部引用）** | 第三方幂等键，如 `extTaskId`、`extVulnId`；平台侧映射内部 ID |
| **Canonical Model（规范模型）** | 平台统一的漏洞实例、任务、状态、事件定义 |
| **Capability（能力项）** | Partner 可启用的 API/事件子集，如仅查询、或含写回 |
| **Compliance Profile（合规档）** | 如 `MIIT-2025`，定义导出字段、码表、工单类型映射 |
| **Connector（连接器）** | 对接 SVMP 等后端系统的适配实现 |

### 2.3 数据流（典型）

```text
Partner 创建任务（带 extTaskId）
  → Gateway 鉴权 + 幂等
  → Domain 创建内部 taskId，调用 SVMP Connector
  → 扫描完成 → 生成/更新 vulInstanceId（规范：vulInfoID）
  → 发布事件 TASK_COMPLETED → 按 Partner 配置 HTTP 回调 / Kafka
Partner 写回验证结果
  → Domain 状态机校验 → SVMP disposal → 可选触发合规导出
```

## 三、规范域模型（平台内部统一语义）

> 第三方不直接依赖部侧表号；通过开放平台枚举与平台映射。监管报送由 Compliance Profile 在出口层组装。

### 3.1 标识体系

| 规范字段 | 层级 | 含义 | 典型来源 |
|----------|------|------|----------|
| `vulId` | 产品漏洞 | 漏洞编号（部侧核准/基线） | 基线库、部侧 |
| `vulInstanceId` | 系统漏洞实例 | 资产上的具体漏洞实例 ID | **平台生成**（对外 API 主键） |
| `externalVulnRef` | 第三方引用 | Partner 侧实例 ID（可选） | Partner 传入 |
| `engineDisposalId` | 引擎内部 | SVMP `vulnDisposalId` 等 | 仅 Connector 使用，不对外 |

**约定**：开放平台写接口以 `vulInstanceId` 为主键；`externalVulnRef` 用于双向关联。

### 3.2 漏洞实例状态 `vulnInstanceStatus`（映射部侧 A.8）

| 码值 | 开放平台枚举 | 说明 | 阶段 |
|------|----------------|------|------|
| 0 | `POTENTIAL` | 潜在预警 | 预警 |
| 1 | `DISCOVERED` | 初始发现 | 识别 |
| 2 | `VERIFIED_VALID` | 已验证有效 | 识别 |
| 3 | `VERIFIED_FALSE_POSITIVE` | 已验证误报 | 修复（终态阻断） |
| 5 | `FIXED` | 已修复 | 修复 |
| 6 | `VERIFY_FIXED_PASS` | 核验修复 | 修复 |
| 7 | `VERIFY_FIXED_FAIL` | 核验未修复 | 识别 |
| 8 | `VERIFY_FAILED` | 验证失败 | 识别 |
| 9 | `FIX_FAILED` | 修复失败 | 识别 |
| 10 | `VERIFY_FIX_FAILED` | 核验失败 | 识别 |

**状态机守卫**（全局，所有 Partner 共用）：

```text
DISCOVERED(1) ──verify──▶ VERIFIED_VALID(2) | VERIFIED_FALSE_POSITIVE(3)
VERIFIED_VALID / VERIFY_FIXED_FAIL(7) ──markFixed──▶ FIXED(5)
未修复态 {1,2,7} ──recordUnfixable──▶ 状态不变 + unfixReason
VERIFIED_FALSE_POSITIVE(3) ──▶ 禁止 markFixed / recordUnfixable
FIXED(5) ──引擎核验──▶ VERIFY_FIXED_PASS(6) | VERIFY_FIXED_FAIL(7)
```

### 3.3 未修复原因 `unfixReason`（映射部侧 A.23 `lvRsn`）

| 码值 | 枚举 | 说明 |
|------|------|------|
| 101–109, 999 | 同部侧 | 仅**未修复态**实例可填写 |

### 3.4 生命周期阶段（与部侧工单对照）

| 平台阶段 | 开放平台能力 | 部侧 orderSubType（MIIT-2025 Profile） |
|----------|--------------|----------------------------------------|
| 排查/发现 | `tasks.*`, `instances.list` | 31/41, 3/4 |
| 验证 | `instances.verify` | 32/42 |
| 修复 | `instances.markFixed` | 33/43 |
| 核验 | `instances.verifyFix`（引擎） | 34/44, 1063 |
| 不可修复备案 | `instances.recordUnfixable` | 无独立状态码，填 `lvRsn` |

## 五、开放平台 API（对外统一契约）

> **Base Path**：`/api/open/v1`  
> **鉴权**：`Authorization: Bearer <accessToken>` 或 `X-Api-Key` + `X-Signature`（HMAC-SHA256）  
> **Partner 识别**：Token 绑定 `partnerId`，路径中不再重复传 partner（管理接口除外）

### 5.1 通用约定

| 项 | 约定 |
|----|------|
| 幂等 | 写接口支持 `Idempotency-Key` 请求头，或 body 内 `extTaskId` / 业务幂等字段 |
| 分页 | `page`（从 1）、`size`（≤1000） |
| 时间 | ISO 8601 UTC，`2026-05-17T08:00:00Z` |
| 响应包装 | `{ "code": 0, "message": "ok", "data": { ... }, "requestId": "..." }` |
| 错误码 | 见 §5.8 |

### 5.2 任务 · 排查/扫描

#### `POST /api/open/v1/tasks`

| 参数 | 类型 | 必填 | 说明 |
|------|------|:---:|------|
| extTaskId | string | ✓ | Partner 幂等键 |
| taskName | string | ✓ | 任务名称 |
| targets | string[] | ✓ | 扫描目标 |
| targetType | enum | ✓ | `IPV4` / `IPV6` / `URL` |
| vulnCategory | enum | ✓ | `SYSTEM` / `WEB` |
| callbackUrl | string | ○ | 覆盖 Partner 默认回调 |
| scanTemplateId | int | ○ | 引擎扫描模板 |
| priority | enum | ○ | `LOW` / `MEDIUM` / `HIGH` |
| scheduleTime | datetime | ○ | 定时执行 |
| options | object | ○ | 端口范围、存活探测、弱口令等 |

**响应 `data`**：`extTaskId`, `taskId`, `status`（`ACCEPTED` / `QUEUED` / `REJECTED`）, `createdAt`

**后置**：新发现实例默认 `vulnInstanceStatus = DISCOVERED`

#### `GET /api/open/v1/tasks/{taskId}`

查询进度，`status`：`PENDING` / `RUNNING` / `FINISHED` / `FAILED`

#### `GET /api/open/v1/tasks`

按 `extTaskId`、`status`、时间范围分页查询（需 `TASK_READ`）

### 5.3 漏洞实例 · 查询

#### `POST /api/open/v1/instances/search`

| 参数 | 类型 | 必填 |
|------|------|:---:|
| taskId | string | ○ |
| extTaskId | string | ○ |
| statusList | enum[] | ○ |
| severityList | enum[] | ○ |
| page, size | int | ✓ |

**列表项（标准字段）**：

`vulInstanceId`, `vulId`, `vulnInstanceStatus`, `unfixReason`, `title`, `severity`, `orgVulnId`, `networkAddress`, `port`, `service`, `exposureType`（`INTERNAL`/`INTERNET`）, `lastTransitionAt`, `extVulnRef`

> 若 Partner 启用 `MIIT-2025` 导出档，可通过 `?exportProfile=MIIT-2025` 或独立导出接口获取部侧字段名版本。

#### `GET /api/open/v1/instances/{vulInstanceId}`

返回规范模型全量字段 + 修复台账 `remediation` + 扩展属性 `extensions`

### 5.4 漏洞实例 · 验证

#### `POST /api/open/v1/instances/{vulInstanceId}/verify`

| 参数 | 必填 | 规则 |
|------|:---:|------|
| extTaskId | ○ | 关联任务 |
| vulnCategory | ✓ | |
| result | ✓ | `VALID` → 2；`FALSE_POSITIVE` → 3 |
| method | ○ | 处置方式码（映射 A.10） |
| evidence | ○ | 证据说明 |
| operator | ✓ | |
| operatedAt | ○ | |

**前置**：`vulnInstanceStatus ∈ { POTENTIAL, DISCOVERED }`（可配置是否允许从 `DISCOVERED` 重验）  
**后置**：`FALSE_POSITIVE` 后拒绝修复/备案类接口

#### `POST /api/open/v1/instances/verify:batch`

批量验证，返回 `success[]` / `failed[]`（部分成功模式）

### 5.5 漏洞实例 · 修复

#### `POST /api/open/v1/instances/{vulInstanceId}/remediate`

| 参数 | 必填 | 说明 |
|------|:---:|------|
| method | ✓ | 如 `PATCH`/`MITIGATION`/`BLOCK`（映射 A.10 1050/1051/1052） |
| description | ✓ | 修复说明 |
| fixLink | 条件 | method=PATCH 时必填 |
| defenseDevice | 条件 | 等效防护/阻断时必填 |
| remediateDuration | ✓ | 如 `"3日"` |
| operator, remediatedAt | ✓ | |

**前置**：未修复态且非误报  
**后置**：`FIXED`，清空 `unfixReason`

### 5.6 漏洞实例 · 不可修复备案

#### `POST /api/open/v1/instances/{vulInstanceId}/unfixable-records`

| 参数 | 必填 |
|------|:---:|
| unfixReason | ✓ |
| reasonDetail | ✓ |
| operator | ✓ |
| approvedBy | ○ |

**后置**：状态不变，写入 `unfixReason`；可选 `archiveId`

### 5.7 事件回调（平台 → Partner）

平台向 Partner `callbackUrl` 投递：

```json
{
  "eventId": "evt-uuid",
  "eventType": "TASK_COMPLETED",
  "occurredAt": "2026-05-17T08:00:00Z",
  "partnerId": "partner-soc-01",
  "payload": {
    "extTaskId": "EXT-2026-0001",
    "taskId": "TASK-111",
    "status": "FINISHED",
    "summary": { "totalInstances": 25, "verifiedValid": 20, "falsePositive": 5 }
  }
}
```

| eventType | 说明 |
|-----------|------|
| `TASK_COMPLETED` | 任务结束 |
| `TASK_FAILED` | 任务失败 |
| `INSTANCE_STATUS_CHANGED` | 实例状态变更 |
| `EXPORT_READY` | 合规导出包就绪（可选） |

**签名**：`X-Webhook-Signature`, `X-Webhook-Timestamp`（5 分钟有效）

Partner 也可选用 **Kafka 订阅**（需单独开通），Topic 按 `partnerId` 隔离。

### 5.8 统一错误码

| code | 含义 |
|------|------|
| 0 | 成功 |
| 40001 | 参数校验失败 |
| 40002 | 状态机不允许 |
| 40003 | 资源不存在 |
| 40004 | 码表/枚举非法 |
| 40101 | 鉴权失败 |
| 40301 | 能力未开通 |
| 40901 | 幂等冲突 |
| 42901 | 限流 |
| 50001 | 引擎（SVMP）调用失败 |
| 50002 | 回调投递失败（异步重试中） |

## 六、协议适配与字段映射

### 6.1 适配器类型

| 适配器 | 方向 | 用途 |
|--------|------|------|
| **Open API Adapter** | 入站 | 第三方 → 平台（主通道） |
| **Webhook Adapter** | 出站 | 平台 → 第三方回调 |
| **Kafka Adapter** | 双向可选 | 高吞吐、多订阅 |
| **Batch Export Adapter** | 出站 | SFTP/对象存储批量文件 |
| **Regulatory Ingress Adapter** | 入站 | 部侧工单/指令解析（若平台承担报送中枢） |

### 6.2 映射配置（Mapping Profile）示例

```yaml
profileId: partner-soc-default
partnerType: SOC
inbound:
  extTaskId: $.socTaskId          # 兼容 V1.1 字段名
  targets: $.targets
outbound:
  listItem:
    vulInstanceId: $.vulInfoID
    vulnInstanceStatus: $.vulInfoStat
    unfixReason: $.lvRsn
    networkAddress: $.vulNetAddr
complianceExport:
  MIIT-2025:
    vulInfoLst:                     # 部侧表34 组装规则
      vulInfoID: $.vulInstanceId
      vulInfoStat: $.vulnInstanceStatus
      lvRsn: $.unfixReason
```

新增 Partner 时优先**配置映射**，避免新增代码分支；仅当协议差异过大时新增 Adapter 实现。

## 七、合规导出（Compliance Profile）

合规不是第三方 API 的形状，而是**出口转换层**：

```text
Canonical Instance  ──▶  Compliance Exporter  ──▶  部侧 vulInfoLst / 工单回传包
                              ▲
                         MIIT-2025 Profile
```

### 7.1 MIIT-2025 Profile（内置）

- 字段、长度、嵌套结构遵循 2025 接口规范表34
- `orderSubType` 与平台阶段映射见 §3.4
- 验收用例继承 V1.1 §八（检测要点 T-02～T-15），改为对 **Exporter 输出** 校验

### 7.2 扩展其他 Profile

未来可增 `CVE-JSON`、`STIX` 等行业格式，与 Partner 的 `complianceProfiles` 绑定，不影响开放平台核心 API。

## 八、安全与治理

| 维度 | 要求 |
|------|------|
| 鉴权 | API Key / OAuth2 Client Credentials；敏感接口强制 HMAC |
| 授权 | 按 Partner `capabilities` 细粒度控制 |
| 审计 | 记录 partnerId、requestId、接口、实例 ID、状态变更前后值 |
| 限流 | 按 Partner 独立配额；写接口可更低阈值 |
| 数据隔离 | 实例查询默认仅返回该 Partner 创建或授权范围内的任务/实例 |
| 密钥轮换 | 支持双密钥过渡期 |

| **P2** | 控制台：Partner 管理、调用统计、失败重试 | 可运营 |
