# 网络安全漏洞管理平台 · 第三方开放平台接口文档

| 项 | 内容 |
|----|------|
| 文档版本 | **V1.0**（对应平台集成方案 V2.3.3） |
| 协议 | HTTPS · REST JSON |
| Base Path | `/api/open/v1` |
| 机器可读规格 | [`openapi/v1/openapi.yaml`](../openapi/v1/openapi.yaml)（OpenAPI 3.1） |
| 适用对象 | 已签约并完成 Partner 注册的第三方系统（SIEM、ITSM、资产平台、安全运营系统等） |

## 1. 概述

### 1.1 能力范围

第三方通过**开放平台 API** 可完成：

| 能力 | 说明 |
|------|------|
| 扫描/排查 | 创建任务、查询进度与结果摘要 |
| 漏洞实例 | 查询、验证、修复、备案、修复核验 |
| 任务结束数据外发 | 任务完成后下载结构化结果包（推荐四阶段 XML） |
| 事件通知 | 平台向 Partner 回调 URL 推送任务/实例/外发就绪事件 |

扫描与漏洞检测由平台内部对接执行引擎完成；Partner **不直接调用**引擎接口。

### 1.2 集成架构

```text
┌─────────────────────┐         HTTPS REST          ┌──────────────────────────┐
│  第三方系统 Partner  │ ◀────────────────────────▶ │  漏洞管理平台 · 开放平台   │
│  (您的系统)          │         Webhook POST         │  /api/open/v1            │
└─────────────────────┘ ◀────────────────────────── └──────────────────────────┘
```

### 1.3 漏洞实例生命周期（写接口）

```text
排查（POST /tasks）
    → 实例默认 vulInfoStat = 1（初始发现）
验证（POST /instances/{vulInfoID}/verify）
    → 2 已验证有效  |  3 已验证误报（终态，不可再处置）
处置（二选一）
    → POST .../remediate  → vulInfoStat = 5（已修复）
    → POST .../archive    → vulInfoStat = 9（修复失败/备案）
修复核验（POST .../verify-fix）
    → 6 核验修复  |  7 核验未修复  |  10 核验失败
```

## 3. 鉴权与安全

### 3.1 方式一：Bearer Token（推荐）

```http
Authorization: Bearer <accessToken>
Content-Type: application/json
```

### 3.2 方式二：API Key + 签名

| 请求头 | 说明 |
|--------|------|
| `X-Api-Key` | 平台分配的 Key |
| `X-Signature` | 对规范串做 **HMAC-SHA256** 的十六进制摘要 |
| `X-Timestamp` | Unix 时间戳（秒），有效期建议 ≤ 5 分钟 |

签名原文（示例，以运营文档为准）：

```text
{HTTP_METHOD}\n{PATH}\n{TIMESTAMP}\n{SHA256_HEX(BODY)}
```

### 3.3 Webhook 验签（Partner 侧实现）

平台向您的 `callbackUrl` 投递事件时携带：

| 请求头 | 说明 |
|--------|------|
| `X-Webhook-Signature` | 事件 body 的 HMAC-SHA256 |
| `X-Webhook-Timestamp` | Unix 秒，5 分钟内有效 |

收到后请先验签再处理业务；建议响应 HTTP `200` 及 `{"received":true}`。

## 4. 通用约定

### 4.1 响应包装

除文件下载外，接口在 **HTTP 200** 时仍使用业务码 `code` 区分成败：

```json
{
  "code": 0,
  "message": "ok",
  "requestId": "req-20260518-001",
  "data": { }
}
```

| 字段 | 说明 |
|------|------|
| `code` | `0` 成功；非 0 见 §9 |
| `message` | 描述信息 |
| `data` | 成功时业务数据；失败多为 `null` |
| `requestId` | 排障追踪 ID，建议日志留存 |

### 4.2 幂等

| 场景 | 约定 |
|------|------|
| 创建任务 | **必填** `extTaskId`（Partner 侧唯一）；重复提交返回 `40901` 或 `200` 且返回已有 `taskId` |
| 查询任务进度 | 仅使用平台返回的 **`taskId`**，不需传 `extTaskId` |
| 实例写操作 | 建议请求头 `Idempotency-Key: {vulInfoID}-{动作}` |

### 4.3 分页与时间

| 项 | 约定 |
|----|------|
| 分页 | `page` 从 1 起；`size` 最大 1000 |
| 时间 | ISO 8601 UTC，如 `2026-05-18T08:00:00Z` |
| 实例 `transferTime` | 部侧常见为 **Unix 秒字符串** |

### 4.4 字段命名

漏洞实例字段与部侧表34 **同名**，例如：`vulInfoID`、`vulInfoStat`、`vulName`、`vulNetAddr`、`remedDesc`、`srcMethod` 等。第三方对接报送系统时可减少字段转换。

### 5.1 创建扫描任务

**`POST /tasks`**

**请求体（主要字段）**

| 参数 | 类型 | 必填 | 说明 |
|------|------|:---:|------|
| extTaskId | string | ✓ | Partner 幂等键 |
| taskName | string | ✓ | 任务名称 |
| targets | string[] | ✓ | 扫描目标 |
| targetType | string | ✓ | `IPV4` / `IPV6` / `URL` |
| vulnType | int | ✓ | `1` 系统漏洞；`2` Web 漏洞 |
| scanTemplateId | int | ○ | 扫描模板 ID |
| exportTemplateId | string | ○ | 外发模板 ID（缺省按模板匹配规则） |
| callbackUrl | string | ○ | 覆盖默认回调地址 |
| options.isLiveProbe | bool | ○ | 主机存活探测 |
| options.portScope | string | ○ | 端口范围，如 `1-65535` |
| options.pswdGuessEnabled | bool | ○ | 弱口令扫描 |

**响应 data**

| 参数 | 说明 |
|------|------|
| taskId | 平台任务 ID（后续查进度、外发均使用） |
| extTaskId | 回显 |
| status | `ACCEPTED` / `QUEUED` / `REJECTED` |
| createdAt | 创建时间 |

**请求示例**

```http
POST /api/open/v1/tasks HTTP/1.1
Authorization: Bearer <token>
Content-Type: application/json

{
  "extTaskId": "EXT-TASK-2026-0001",
  "taskName": "2026Q2-核心系统排查",
  "targets": ["10.10.1.1", "10.10.1.2"],
  "targetType": "IPV4",
  "vulnType": 1,
  "scanTemplateId": 10086,
  "exportTemplateId": "tpl-svmp-xml-scan-bundle",
  "options": {
    "isLiveProbe": true,
    "portScope": "1-65535",
    "pswdGuessEnabled": true
  }
}
```

**成功响应示例**

```json
{
  "code": 0,
  "message": "ok",
  "requestId": "req-20260518-t001",
  "data": {
    "extTaskId": "EXT-TASK-2026-0001",
    "taskId": "TASK-7f3a2b1c",
    "status": "ACCEPTED",
    "createdAt": "2026-05-18T08:00:00Z"
  }
}
```

### 5.2 查询任务进度

**`GET /tasks/{taskId}`**

| 响应 data 字段 | 说明 |
|----------------|------|
| status | `QUEUED` / `RUNNING` / `FINISHED` / `FAILED` 等 |
| progress | 0–100 |
| startedAt / finishedAt | 起止时间 |
| errorMessage | 失败原因（如有） |

---

### 5.3 查询漏洞实例

**`POST /instances/search`**

常用筛选：

| 参数 | 说明 |
|------|------|
| taskId | 按任务过滤 |
| vulInfoStatList | 状态列表，如 `[1,2,5]` |
| vulLevelList | 危害等级 |
| page / size | 分页 |
| exportProfile | 可选 `MIIT-2025`，返回部侧字段名版本 |

**`GET /instances/{vulInfoID}`** — 单条详情（含 `remedInfo`、组件信息等）。

---

### 5.4 验证

**`POST /instances/{vulInfoID}/verify`**

| 参数 | 必填 | 说明 |
|------|:---:|------|
| result | ✓ | `VALID`（→2）或 `FALSE_POSITIVE`（→3） |
| comment | ○ | 备注 |

前置：`vulInfoStat` 须为 `1`。误报（3）后不可再修复/备案。

---

### 5.5 处置 · 修复

**`POST /instances/{vulInfoID}/remediate`**

| 参数 | 必填 | 说明 |
|------|:---:|------|
| remedInfo.srcMethod | ✓ | 处置方式码（如 `1050` 补丁修复） |
| remedInfo.remedDesc | ○ | 修复说明 |
| remedInfo.fixLnk | 条件 | `srcMethod=1050` 时建议填写补丁链接 |

成功：`vulInfoStat` → **5**。

---

### 5.6 处置 · 备案

**`POST /instances/{vulInfoID}/archive`**

| 参数 | 必填 | 说明 |
|------|:---:|------|
| lvRsn | ✓ | 未修复原因码（部侧 A.23） |
| remedInfo.remedDesc | ○ | 说明 |

成功：`vulInfoStat` → **9**。与修复分支互斥（已修复后不可备案）。

---

### 5.7 修复核验

**`POST /instances/{vulInfoID}/verify-fix`**

| 参数 | 必填 | 说明 |
|------|:---:|------|
| verifyMethod | ○ | 核验方式（如复扫） |
| comment | ○ | 备注 |

前置：`vulInfoStat` 须为 **5**。结果：`6` / `7` / `10`。

> 与 **`/verify`（验证有效/误报）** 不同：后者用于排查阶段确认漏洞真实性。

---

### 5.8 任务结束 · 外发包查询与下载

任务扫描结束且外发组装完成后，平台推送 **`EXPORT_READY`** 事件（§6），Partner 可拉取文件。

#### 查询外发元数据

**`GET /exports/{exportId}`**

| 响应 data 字段 | 说明 |
|----------------|------|
| exportId | 外发记录 ID |
| taskId / extTaskId | 关联任务 |
| exportTemplateId | 使用的模板 |
| format | `JSON` / `CSV` / `SVMP-TAR-GZ-XML` / `MIIT-2025-VULINFOLST` 等 |
| status | `READY` / `EXPIRED` / `FAILED` |
| recordCount | 记录条数 |
| expiresAt | 下载过期时间 |
| downloadUrl | 可选预签名 URL（短期有效） |

#### 下载外发文件

**`GET /exports/{exportId}/download`**

- 响应：`Content-Type` 为 `application/gzip` 或 `application/json` 等，依 `format` 而定。
- 推荐格式 **`SVMP-TAR-GZ-XML`**：解压后为四阶段 XML（§7）。

#### 任务下外发历史

**`GET /tasks/{taskId}/exports`** — 返回该任务历次 `exportId` 列表。

## 6. 事件回调（Webhook）

平台向 Partner 注册的 URL 发起 **`POST`**，`Content-Type: application/json`。

### 6.1 公共请求体

| 字段 | 类型 | 说明 |
|------|------|------|
| eventId | string | 事件唯一 ID（幂等处理） |
| eventType | string | 见下表 |
| occurredAt | datetime | 事件发生时间 |
| partnerId | string | 接入方 ID |
| payload | object | 随事件类型变化 |

### 6.2 事件类型

| eventType | 说明 |
|-----------|------|
| `TASK_COMPLETED` | 任务正常结束 |
| `TASK_FAILED` | 任务失败 |
| `INSTANCE_STATUS_CHANGED` | 实例状态变更 |
| `INSTANCE_REMEDIATED` | 修复完成（→5） |
| `INSTANCE_ARCHIVED` | 备案完成（→9） |
| `INSTANCE_VERIFY_FIX_COMPLETED` | 修复核验完成 |
| `EXPORT_READY` | 外发包可下载 |

### 6.3 示例：任务完成

```json
{
  "eventId": "evt-20260518-0001",
  "eventType": "TASK_COMPLETED",
  "occurredAt": "2026-05-18T14:00:00Z",
  "partnerId": "partner-demo-01",
  "payload": {
    "extTaskId": "EXT-TASK-2026-0001",
    "taskId": "TASK-7f3a2b1c",
    "status": "FINISHED",
    "summary": {
      "totalInstances": 25,
      "verifiedValid": 18,
      "falsePositive": 5
    }
  }
}
```

### 6.4 示例：外发就绪

```json
{
  "eventId": "evt-20260518-0002",
  "eventType": "EXPORT_READY",
  "occurredAt": "2026-05-18T14:05:00Z",
  "partnerId": "partner-demo-01",
  "payload": {
    "exportId": "EXP-20260518-7f3a",
    "taskId": "TASK-7f3a2b1c",
    "extTaskId": "EXT-TASK-2026-0001",
    "exportTemplateId": "tpl-svmp-xml-scan-bundle",
    "format": "SVMP-TAR-GZ-XML",
    "recordCount": 128,
    "downloadUrl": "https://vuln-platform.example.com/download/EXP-20260518-7f3a?sig=..."
  }
}
```

非 2xx 响应将触发平台重试（策略以运营配置为准）。

## 7. 任务结束数据外发（四阶段 XML）

### 7.1 包结构

推荐模板 **`tpl-svmp-xml-scan-bundle`**，产出 **`tar.gz`**，按任务启用的扫描能力选择性包含：

| 文件 | 扫描阶段 | 根元素 | 说明 |
|------|----------|--------|------|
| `主机存活探测结果.xml` | 存活探测 | 推导结构 | 待引擎正式样本补齐 |
| `端口扫描结果.xml` | 端口扫描 | 推导结构 | 待引擎正式样本补齐 |
| `系统漏洞扫描结果.xml` | 系统漏洞 | `<aurora>` | 绿盟 Aurora 报告格式 |
| `弱口令扫描结果.xml` | 弱口令 | `<aurora>` | `task_type=4`，含 `password_results` |

### 7.2 Aurora 报告结构（系统漏洞 / 弱口令）

```text
aurora
├── ret_code / ret_msg
└── data
    └── report
        ├── vendor, product, version
        ├── task (id, name, task_type, time_start_scan, time_end_scan)
        ├── scanned_ip_count
        └── targets / target
            ├── ip, host_score, vul_score
            ├── vuln_scanned / vuln      # 目标上的漏洞实例
            ├── vuln_detail / vuln       # 漏洞详情（name, cve_id, cnnvd, risk_points…）
            ├── password_results         # 弱口令（type, username, password）
            └── appendix_info            # 端口 Banner、远程端口信息等
```

字段与平台规范域映射见仓库 `export-templates/xml-schemas/aurora-report.schema.json`。

**安全**：弱口令 XML 中 `password` 字段在对外投递前须脱敏；Partner 存储与展示需符合本单位数据安全要求。

### 7.3 获取方式

1. 订阅 `EXPORT_READY` → 使用 `downloadUrl` 或 `GET /exports/{exportId}/download`
2. 或轮询 `GET /tasks/{taskId}/exports` 直至出现 `status=READY`

| 能力码 | 说明 |
|--------|------|
| `TASK_WRITE` | 创建任务 |
| `TASK_READ` | 查询任务 |
| `INSTANCE_READ` | 查询实例 |
| `INSTANCE_VERIFY` | 验证 |
| `INSTANCE_REMEDIATE` | 修复 |

## 9. 业务错误码

| code | 含义 | 处理建议 |
|------|------|----------|
| 0 | 成功 | — |
| 40001 | 参数校验失败 | 检查请求体 |
| 40002 | 状态机不允许 | 核对当前 `vulInfoStat` |
| 40003 | 资源不存在 | 检查 ID |
| 40004 | 枚举/码表非法 | 对照附录状态表 |
| 40005 | 修复与备案互斥 | 勿重复处置 |
| 40101 | 鉴权失败 | 检查 Token/签名 |
| 40301 | 能力未开通 | 联系运营开通 |
| 40901 | 幂等冲突 | 使用已返回的 `taskId` |
| 42901 | 限流 | 退避重试 |
| 50001 | 引擎调用失败 | 稍后重试或联系平台 |
| 50002 | 回调投递失败 | 平台异步重试；检查 Partner 接收端 |

## 10. 典型集成流程

```text
1. POST /tasks（extTaskId 幂等）→ 保存 taskId
2. 轮询 GET /tasks/{taskId} 或等待 Webhook TASK_COMPLETED
3. 收到 EXPORT_READY → GET /exports/{exportId}/download → 解析 tar.gz / XML
4. POST /instances/search?taskId=... → 入库漏洞实例
5. 业务侧处置后：
   - POST .../verify（有效/误报）
   - POST .../remediate 或 .../archive
   - POST .../verify-fix
6. 可选：订阅 INSTANCE_* 事件驱动 ITSM 工单
```
