# 开放平台 P0 联调手册

> **链路（推荐）**：partner-gateway 校验 Token → 注入 `X-Partner-Id` → open-api-service 任务 API → vul-pass 执行  
> **范围**：P0 任务读/写 + Partner 隔离 + open-api-service 自管 Partner/Token；不含 P1 Webhook / 实例写。  
> **对照**：[组件职责与接口映射](./组件职责与接口映射.md) · [Partner 鉴权落地方案 §8](./开放平台Partner鉴权与隔离-落地方案.md)

## open-api 业务（P0 · 任务域）

### A. 服务与端口

| 服务 | Nacos 名 | 默认端口 | 说明 |
|------|----------|----------|------|
| partner-gateway | `partner-gateway` | （按部署） | 公网入口；Token 校验 + 注入 `X-Partner-Id` |
| open-api-service | `open-api-service` | **35780** | Partner/Token + 任务 API |
| vul-pass | `vul-pass` | （按部署） | SVMP 扫描执行（真实 Feign 目标） |

### B. SVMP 接口对照（SvmpEngineAdapter）

| 开放平台动作 | 逻辑契约（设计 §6.3） | vul-pass **真实** REST | 入参要点 |
|--------------|----------------------|------------------------|----------|
| 创建任务 | `POST /task/create` | `POST /vul-scan-task/dispatch` | 扫描目标 → `assetList`；**不带** `extTaskId` |
| 查进度 | `GET /task/progress?taskId=` | `GET /vul-scan-task/page2?id=` | **仅** vul-pass 主任务 `id`（平台侧 `engineTaskId`） |

配置（Nacos / `application.yml`）：

```yaml
open-api:
  svmp:
    engine-service-name: vul-pass
    engine-path-prefix: ""
    dispatch:
      order-id: "1-31-0000000000000000001"   # 须在 vul-pass 指令库存在
      tsk-phase: 1
      proc-method: 1022
      eng-hashes: [NA]
```

### C. curl 步骤（两家 Partner 互不可见）

```bash
BASE=http://127.0.0.1:35780
ADMIN_KEY=dev-internal-admin-key-change-in-prod
PARTNER_A=partner-demo-01
PARTNER_B=partner-demo-02
EXT_ID=EXT-$(date +%s)
```

#### C.1 创建 Partner A / B（内网管理 API）

```bash
curl -s -X POST "${BASE}/internal/admin/partners" \
  -H "Content-Type: application/json" \
  -H "X-Internal-Admin-Key: ${ADMIN_KEY}" \
  -d '{"partnerId":"'"${PARTNER_A}"'","partnerName":"Demo A","partnerType":"SIEM","capabilities":["TASK_WRITE","TASK_READ"]}'

curl -s -X POST "${BASE}/internal/admin/partners" \
  -H "Content-Type: application/json" \
  -H "X-Internal-Admin-Key: ${ADMIN_KEY}" \
  -d '{"partnerId":"'"${PARTNER_B}"'","partnerName":"Demo B","partnerType":"SIEM","capabilities":["TASK_WRITE","TASK_READ"]}'
```

#### C.2 创建凭证并换 Token（Partner A）

```bash
CRED=$(curl -s -X POST "${BASE}/internal/admin/partners/${PARTNER_A}/credentials" \
  -H "X-Internal-Admin-Key: ${ADMIN_KEY}")
CLIENT_ID=$(echo "${CRED}" | sed -n 's/.*"clientId"[[:space:]]*:[[:space:]]*"\([^"]*\)".*/\1/p' | head -1)
CLIENT_SECRET=$(echo "${CRED}" | sed -n 's/.*"clientSecret"[[:space:]]*:[[:space:]]*"\([^"]*\)".*/\1/p' | head -1)

TOKEN_RESP=$(curl -s -X POST "${BASE}/oauth/token" \
  -H "Content-Type: application/json" \
  -d '{"grantType":"client_credentials","clientId":"'"${CLIENT_ID}"'","clientSecret":"'"${CLIENT_SECRET}"'"}')
echo "${TOKEN_RESP}"
```

经 **partner-gateway** 时：同上路径，Header 带 `Authorization: Bearer {accessToken}`，网关剥离 Token 并注入 `X-Partner-Id`。

#### C.3 缺 X-Partner-Id → 40001（内网直调说明）

直连 open-api-service **必须**手动带头（生产禁止绕过网关）：

```bash
curl -s "${BASE}/api/open/v1/tasks?page=1&size=10"
```

**预期**：`code: 40001`，`message: 缺少请求头 X-Partner-Id`。

#### C.4 Partner A 创建任务（TASK_WRITE）

```bash
CREATE_A=$(curl -s -X POST "${BASE}/api/open/v1/tasks" \
  -H "X-Partner-Id: ${PARTNER_A}" \
  -H "Content-Type: application/json" \
  -d '{"extTaskId":"'"${EXT_ID}"'","taskName":"p0-open-api","targets":["10.0.0.1"],"targetType":"IPV4","vulnType":1}')
echo "${CREATE_A}"
TASK_ID=$(echo "${CREATE_A}" | sed -n 's/.*"taskId"[[:space:]]*:[[:space:]]*"\([^"]*\)".*/\1/p' | head -1)
```

**链路**：写 `open_task` / `partner_task_map` → Feign `POST /vul-scan-task/dispatch` → 回写 `engine_task_id`。  
vul-pass 不可用或未配置 `dispatch.order-id` 时：**`code: 50001`**。

#### C.5 幂等 40901

重复 C.4 相同 `extTaskId` → **`code: 40901`**，`data` 含已有 `taskId`。

#### C.6 Partner B 访问 A 的 taskId → 40003

```bash
curl -s "${BASE}/api/open/v1/tasks/${TASK_ID}" -H "X-Partner-Id: ${PARTNER_B}"
```

**预期**：`code: 40003`，`message: 无权访问该任务`。

#### C.7 查进度 / 列表（TASK_READ）

```bash
curl -s "${BASE}/api/open/v1/tasks/${TASK_ID}" -H "X-Partner-Id: ${PARTNER_A}"
curl -s "${BASE}/api/open/v1/tasks?page=1&size=10" -H "X-Partner-Id: ${PARTNER_A}"
curl -s "${BASE}/api/open/v1/tasks?page=1&size=10" -H "X-Partner-Id: ${PARTNER_B}"
```

Partner B 列表中**不得**出现 Partner A 任务。

#### C.8 一键脚本

```bash
export BASE=http://127.0.0.1:35780 PARTNER_A=partner-demo-01 PARTNER_B=partner-demo-02
bash project_backend/svmp/open-api-service/scripts/curl-verify.sh
bash project_backend/svmp/open-api-service/scripts/curl-partner-auth.sh
```

## 1. 集成常量（三模块必须一致）

| 项 | 约定 | clover | morningglory | open-api-service |
|----|------|--------|--------------|------------------|
| Redis Token 键 | `partner:token:{sha256(accessToken)}` | 写入 | 读取（默认 `introspect-mode: redis`） | 不读 Token |
| Token JSON 字段 | `subjectType`,`partnerId`,`capabilities`,`clientId`,`issuedAt`,`expiresAt` | `PartnerTokenContextDTO` | `PartnerTokenContext` | — |
| `subjectType` | 固定 `PARTNER` | 写入 | 校验 | — |
| Partner 上下文头 | `X-Partner-Id` | — | 注入 | **唯一**可信来源 |
| 追踪头 | `X-Request-Id` | — | 生成/透传 | 写入响应 `requestId` |
| 能力码 | 全大写，见 §8 | DB `partner_capability` | 路径拦截 | — |
| 开放平台错误体 | HTTP **200** + JSON | Token 端点除外 | `{code,message,data,requestId}` | 同左 |

> **注意**：Partner 鉴权落地方案 §3 示例键 `token:sha256(...)` 为旧描述；**以实现为准**：`partner:token:` 前缀。

## 2. 环境准备

### 2.1 服务与端口（本地示例）

| 服务 | Nacos 名 | 默认端口 | 说明 |
|------|----------|----------|------|
| morningglory | `morningglory` | 7000 | 公网入口 |
| clover | `clover` | （按部署） | Token 签发 + Partner 管理 |
| open-api-service | `open-api-service` | 35780 | 开放平台业务 |
| SVMP 引擎 | `vul-pass`（可配） | — | `POST /vul-scan-task/dispatch`、`GET /vul-scan-task/page2` |

### 2.2 morningglory 必开配置

```yaml
gateway:
  open-api:
    enabled: true
    path-prefix: /api/open/v1
    token-paths: /oauth/token,/api/open/v1/oauth/token
    introspect-mode: redis   # 与 clover 共用 Redis
    strip-authorization: true
```

Nacos 路由（`StripPrefix=0`）：

- `/api/open/v1/**` → `lb://open-api-service`
- `/oauth/token`、`/api/open/v1/oauth/token` → `lb://clover`

### 2.3 Redis / DB

- clover 与 morningglory **共用** Partner Token Redis（键前缀 `partner:token:`）。
- open-api-service 库：`open_task`、`partner_task_map`（Liquibase 见 `open-api-service/src/main/resources/db/mysql/`）。

## 3. 联调步骤（curl）

以下变量请按环境替换：

```bash
GW=http://127.0.0.1:7000          # morningglory
CLOVER=http://127.0.0.1:8080      # 直连 clover（仅管理步骤）
PARTNER_A=partner-demo-01
PARTNER_B=partner-demo-02
EXT_ID=EXT-$(date +%s)
```

### 步骤 0 · 健康检查

```bash
curl -s http://127.0.0.1:35780/internal/health
```

**预期**：HTTP 200，服务可达。

---

### 步骤 1 · 创建 Partner A（clover 内部接口）

```bash
curl -s -X POST "${CLOVER}/internal/partners" \
  -H "Content-Type: application/json" \
  -d '{
    "partnerId":"'"${PARTNER_A}"'",
    "partnerName":"Demo Partner A",
    "partnerType":"SIEM",
    "capabilities":["TASK_WRITE","TASK_READ"]
  }'
```

**预期**：`success: true`；capabilities 入库为全大写 `TASK_WRITE`,`TASK_READ`。

对 Partner B 重复，`partnerId` 改为 `partner-demo-02`。

---

### 步骤 2 · 创建 client 凭证

```bash
curl -s -X POST "${CLOVER}/internal/partners/${PARTNER_A}/credentials"
```

**预期**：返回 `clientId`（`cli_...`）与 **`clientSecret`（仅一次）**，请保存：

```bash
CLIENT_ID=cli_xxxxxxxx
CLIENT_SECRET=xxxxxxxx
```

---

### 步骤 3 · 取 Token（经网关 → clover）

```bash
curl -s -X POST "${GW}/oauth/token" \
  -H "Content-Type: application/json" \
  -d '{
    "grant_type":"client_credentials",
    "client_id":"'"${CLIENT_ID}"'",
    "client_secret":"'"${CLIENT_SECRET}"'"
  }'
```

**预期（OAuth 标准 snake_case，HTTP 200）**：

```json
{
  "access_token": "eyJ...",
  "token_type": "Bearer",
  "expires_in": 86400,
  "partner_id": "partner-demo-01"
}
```

**失败示例**（错误凭证）：HTTP 401，clover `ErrorResult`（非开放平台 `40101` 包装）。

保存 Token：

```bash
TOKEN=<access_token 值>
```

**Redis 校验（可选）**：

```bash
HASH=$(printf '%s' "${TOKEN}" | sha256sum | awk '{print $1}')
redis-cli GET "partner:token:${HASH}"
```

**预期**：JSON 含 `"subjectType":"PARTNER"`、`"partnerId":"partner-demo-01"`、`"capabilities":["TASK_WRITE","TASK_READ"]`。

---

### 步骤 4 · 网关鉴权失败（40101）

```bash
curl -s "${GW}/api/open/v1/tasks"
```

**预期**：

```json
{
  "code": 40101,
  "message": "Partner token invalid or expired",
  "data": null,
  "requestId": "req-..."
}
```

HTTP 状态码：**200**。

---

### 步骤 5 · 能力未开通（40301）

先写入**仅有 TASK_READ** 的 Token（或创建只读 Partner），然后：

```bash
curl -s -X POST "${GW}/api/open/v1/tasks" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"extTaskId":"t1","taskName":"x","targets":["10.0.0.1"],"targetType":"IPV4","vulnType":1}'
```

**预期**：

```json
{
  "code": 40301,
  "message": "Capability not granted: TASK_WRITE",
  "data": null,
  "requestId": "req-..."
}
```

---

### 步骤 6 · 创建任务（TASK_WRITE + 引擎）

```bash
curl -s -X POST "${GW}/api/open/v1/tasks" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "extTaskId":"'"${EXT_ID}"'",
    "taskName":"p0-curl-test",
    "targets":["10.0.0.1"],
    "targetType":"IPV4",
    "vulnType":1
  }'
```

**预期（HTTP 200）**：

```json
{
  "code": 0,
  "message": "ok",
  "requestId": "req-...",
  "data": {
    "extTaskId": "EXT-...",
    "taskId": "TASK-...",
    "status": "ACCEPTED",
    "createdAt": "2026-05-21T..."
  }
}
```

**链路确认**：

1. morningglory 注入 `X-Partner-Id: partner-demo-01`，剥离 `Authorization`
2. open-api-service 写 `open_task` / `partner_task_map`
3. Feign 调 vul-pass `POST /vul-scan-task/dispatch`（**不带** `extTaskId`）
4. 下发成功后 `GET /vul-scan-task/page2?orderId=` 解析 `engine_task_id` 写入 `open_task`

引擎不可用时的 **预期**：`code: 50001`，`message: 引擎创建任务失败`。

---

### 步骤 7 · 幂等冲突（40901）

重复步骤 6 相同 `extTaskId`。

**预期**：

```json
{
  "code": 40901,
  "message": "extTaskId 已存在",
  "data": { "taskId": "TASK-...", "extTaskId": "...", ... }
}
```

---

### 步骤 8 · 查进度 / 列表（TASK_READ）

```bash
TASK_ID=<步骤6返回的 taskId>

curl -s "${GW}/api/open/v1/tasks/${TASK_ID}" \
  -H "Authorization: Bearer ${TOKEN}"

curl -s "${GW}/api/open/v1/tasks?page=1&size=10" \
  -H "Authorization: Bearer ${TOKEN}"
```

**预期**：`code: 0`；`data` 含任务进度或分页列表；仅本 Partner 数据。

---

### 步骤 9 · 跨 Partner 隔离（40003）

Partner B 取 Token 后，访问 Partner A 的 `taskId`：

```bash
curl -s "${GW}/api/open/v1/tasks/${TASK_ID}" \
  -H "Authorization: Bearer ${TOKEN_B}"
```

**预期**：

```json
{
  "code": 40003,
  "message": "无权访问该任务",
  "data": null,
  "requestId": "req-..."
}
```

Partner B 列表接口不应出现 Partner A 任务。

---

### 步骤 10 · 直连 open-api-service（跳过网关，仅开发）

```bash
export BASE=http://127.0.0.1:35780
export PARTNER_A=partner-demo-01
export PARTNER_B=partner-demo-02
bash project_backend/svmp/open-api-service/scripts/curl-verify.sh
```

手动注入 `X-Partner-Id`；**不可**用于生产验收。

## 5. 错误码速查（P0）

| code | 产生组件 | 典型场景 |
|------|----------|----------|
| 0 | open-api-service | 成功 |
| 40001 | open-api-service | 参数校验 / 直连缺 `X-Partner-Id` |
| 40003 | open-api-service | 跨 Partner 访问 `taskId` |
| 40101 | morningglory | Token 无效 / 非 PARTNER 类型 |
| 40301 | morningglory | capability 未开通 |
| 40901 | open-api-service | `(partner_id, ext_task_id)` 冲突 |
| 42901 | morningglory | 限流（`rate-limit.enabled=true` 时） |
| 50001 | open-api-service | SVMP 引擎调用失败 |

## 6. 上线前检查清单（Partner 鉴权 §8）

- [ ] 第三方仅持有 Partner `clientId/secret`，非门户用户密码
- [ ] Redis/introspect 必含 `partnerId`、`subjectType=PARTNER`
- [ ] Redis 键为 `partner:token:{sha256(token)}`（clover 写、morningglory 读）
- [ ] `/api/open/v1` **不**用 body/query 中 `partnerId` 鉴权（仅 `X-Partner-Id`）
- [ ] `extTaskId` 唯一约束为 `(partner_id, ext_task_id)`
- [ ] 跨 Partner 访问他人 `taskId` 返回 **40003**
- [ ] 网关与业务日志打印 `partnerId`、`requestId`，**不**打印 secret
- [ ] Webhook 使用 Partner 级 `webhook_secret`（P1；与 client_secret 分离）
- [ ] `gateway.open-api.enabled=true` 且 Nacos 路由 `/api/open/v1/**` → `open-api-service`
- [ ] 能力码与 §8 一致（全大写：`TASK_WRITE` 等）
- [ ] `/oauth/token` 请求支持 `grant_type`/`client_id`/`client_secret`（snake_case）
- [ ] `/oauth/token` 响应为 `access_token`/`expires_in`/`partner_id`（非 Result 包装）

## 7. 常见问题

| 现象 | 排查 |
|------|------|
| 40101 | Token 未写入 Redis / 过期；morningglory 与 clover Redis 库不一致 |
| 40301 | Partner 未开通对应 capability；确认 DB 为大写 `TASK_WRITE` |
| 40001 缺 X-Partner-Id | 绕过网关直连 open-api-service；生产必须经 morningglory |
| 50001 | SVMP 未注册或 `open-api.svmp.engine-service-name` 错误 |
| Token curl 无 `access_token` | 使用了旧版 Result 包装；应直接解析 OAuth 响应体 |
