## 0. 架构决策与迁移说明

### 0.1 新架构原则（零侵入）

| 原则 | 说明 |
|------|------|
| **现有服务不变** | **morningglory**（门户网关）、**clover**（门户 IAM）**不做** Partner 模块改造 |
| **独立 Partner 入口** | 新建 **partner-gateway**，公网域名如 `open.vuln.example.com`，与门户流量物理隔离 |
| **开放平台自包含** | **open-api-service** 承载 Partner 注册、凭证、Token 签发、全部 `/api/open/v1/*`、Webhook、SVMP 适配 |
| **Partner 不直连 SVMP** | 仅 open-api-service 调引擎接口 |
| **数据隔离** | 只信任 partner-gateway 注入的 **`X-Partner-Id`**；禁止用 body/query 中的 partnerId 鉴权 |
| **extTaskId** | **不下发** SVMP；唯一约束 `(partner_id, ext_task_id)` |

### 0.2 废弃方案（迁移说明）

> ⚠️ **已废弃**：在 **morningglory** 增加 `PartnerOpenApiFilter`、在 **clover** 增加 Partner 表与 `/oauth/token` 的「骨架方案」。  
> 该方案侵入现有门户网关与 IAM，联调与发布耦合门户发版，**不再采用**。

| 旧方案（废弃） | 新方案（当前） |
|----------------|----------------|
| morningglory 路由 `/api/open/v1/**` | **partner-gateway** 独立路由 |
| clover 签发 Partner Token | **open-api-service** 签发 |
| clover 存 Partner / credentials | **open-api-service** 库 |
| morningglory capability / 限流 | **partner-gateway** capability / 限流 |
| clover `/internal/token/introspect` | partner-gateway 查 Redis 或 Feign open-api-service introspect |

存量文档（如 [联调手册-P0](./联调手册-P0.md)、[README §internal](../README.md) 旧版架构图）若仍引用 morningglory/clover Partner 改造，以**本文**为准。

## 1. 总体链路

```text
Partner
  │  HTTPS（公网）
  ▼
partner-gateway ──路由──► open-api-service ──Feign/REST──► SVMP 执行层
  │  Partner 校验              │  Token 签发 / 业务 / 幂等 / Webhook
  │  capability / 限流         │  Partner 注册 / 凭证 / 管理 API
  │  注入 X-Partner-Id         │
  │                            └── Feign（可选）──► clover /v3/cache/*（名称解析）
  │
  └── Token 校验 ◄── Redis（open-api-service 写入）

morningglory ──► 门户 / 内部 API（与 Partner **完全隔离**，⛔ 不变）
clover       ──► 门户 IAM（与 Partner **完全隔离**，⛔ 不变）
```

**强制规则**

| 规则 | 说明 |
|------|------|
| Token 签发 | 仅 **open-api-service**；partner-gateway **不签发** |
| Partner 上下文 | 仅信任 partner-gateway 注入的 **`X-Partner-Id`** |
| SVMP 调用 | 仅 **open-api-service**；Partner **不直连** |
| 门户网关 | **morningglory** 不承载 `/api/open/v1/**`、不校验 Partner Token |
| 门户 IAM | **clover** 不参与 Partner 注册 / 凭证 / Token |

## 2. morningglory · 门户网关（⛔ 不变 / 开放平台 out of scope）

**职责（现有）**：门户 TLS、JWT / CLIENT 鉴权、`/v3/*` 路由、内部服务发现。  
**开放平台**：**不在此组件实现**任何 Partner 路由、过滤器、限流或上下文注入。

| 项 | 状态 | 说明 |
|----|------|------|
| `/api/open/v1/**` 路由 | ⛔ 不添加 | Partner 流量走 **partner-gateway** |
| `PartnerOpenApiFilter` | ⛔ 不实现 | 已废弃 |
| `/oauth/token` Partner 端点 | ⛔ 不添加 | Token 由 open-api-service 提供 |
| 现有 `JWTAuthFilter`、门户路由 | ✅ 保持 | 与开放平台零耦合 |

> Partner 公网入口为独立域名（如 `open.vuln.example.com` → partner-gateway），与 morningglory 门户域名分离，互不影响发版。

## 3. clover · 门户 IAM（⛔ 不变 / Partner 模块 out of scope）

**职责（现有）**：门户用户登录、角色、组织、CLIENT 动态认证等。  
**开放平台**：**不**实现 Partner 注册、凭证、capabilities 配置、Partner Token 签发或 introspect。

### 3.1 Partner 相关（⛔ 全部 out of scope）

| 原计划接口 | 状态 | 迁移至 |
|------------|------|--------|
| `/internal/partners/*` | ⛔ 不实现 | open-api-service `/internal/admin/partners/*` |
| `/oauth/token`（Partner） | ⛔ 不实现 | open-api-service `POST /oauth/token` |
| `/internal/token/introspect` | ⛔ 不实现 | open-api-service `/internal/token/introspect` |
| Partner 数据表 | ⛔ 不建 | open-api-service 库 |

### 3.2 可选 Feign 复用（名称解析）

open-api-service **可按需** Feign 调用 clover 现有缓存接口，**仅**用于操作人 / 组织名称展示，**不用于** Partner 鉴权：

| 前缀 | 控制器 | 用途 |
|------|--------|------|
| `/v3/cache/*` | CloverCacheEventUI | 用户 / 组织 / 资源组名称解析 |
| `/v3/pub/login` | SigninUI | 门户用户登录（与 Partner **隔离**） |
| `/v3/user`、`/v3/role`、`/v3/org` | 各 UI | 门户管理（集成后台若复用门户登录） |

> clover 的 `clover_client_auth_*` 表与 `ClientAuthController` 继续服务旧版 CLIENT Token，**不**扩展为 Partner 模型。

## 4. partner-gateway · Partner 专用网关（🆕）

**职责**：Partner 公网 TLS、路由、Partner Token 校验、capability 拦截、按 Partner 限流、向下游注入上下文。  
**不负责**：Token 签发、Partner 注册、开放平台业务、SVMP 调用、Webhook 出站。

**部署**：独立服务 + 独立公网域名（如 `https://open.vuln.example.com`），与 morningglory **无共享路由配置**。

### 4.1 路由（Nacos 配置 · 🆕）

| 序号 | 路径模式 | 目标服务 | 说明 |
|------|----------|----------|------|
| R1 | `/api/open/v1/**` | `open-api-service` | 开放平台 REST（需 Partner 鉴权） |
| R2 | `/oauth/token` | `open-api-service` | Partner 取 Token（**免** Partner 鉴权，见白名单） |
| R3 | `/api/open/v1/oauth/token` | `open-api-service` | 同上（若采用统一前缀） |
| R4 | `/internal/**` | — | **禁止**公网暴露；管理 API 仅内网或独立 admin 入口 |

> 本网关**仅**服务 Partner 流量；不启用 morningglory 的 discovery locator 或门户 `/v3/*` 路由。

### 4.2 过滤器逻辑（🆕 `PartnerGatewayFilter`）

| 步骤 | 编号 | 动作 | 失败响应 |
|------|------|------|----------|
| 路径分流 | G1 | 白名单路径跳过 Partner 鉴权；其余 `/api/open/v1/**` 启用 | — |
| Token 校验 | G2 | Redis `partner:token:{hash}` 或 Feign open-api-service `/internal/token/introspect` | HTTP 200, `code=40101` |
| 主体类型 | G3 | `subjectType != PARTNER` → 拒绝 | `40101` |
| 能力匹配 | G4–G5 | 查「方法 + 路径 → capability」表 | `40301` |
| 上下文注入 | G6 | 设置 `X-Partner-Id`、`X-Request-Id`（可选 `X-Capabilities`） | — |
| 限流 | G7 | 按 `partnerId` QPS（Partner 表 `rate_limit_qps` 或 Sentinel） | `42901` |
| HMAC（P2） | G8 | `X-Api-Key` + `X-Signature` + `X-Timestamp` 校验 | `40101` |
| IP / mTLS（P3） | G9 | 连接层策略 | `40101` |

**建议过滤器顺序**：`IgnoreURL`（白名单）→ `PartnerGatewayFilter`（G1–G9）→ 下游路由。

Base Path：`/api/open/v1`。与 [OpenAPI](../../openapi/v1/openapi.yaml) 及 [Partner 鉴权 §4.2](./开放平台Partner鉴权与隔离-落地方案.md) 一致。

| GET | `/instances/{vulInfoID}` | `INSTANCE_READ` | P0 |
| POST | `/instances/{vulInfoID}/verify` | `INSTANCE_VERIFY` | P1 |
| POST | `/instances/verify:batch` | `INSTANCE_VERIFY` | P1 |
| POST | `/instances/{vulInfoID}/remediate` | `INSTANCE_REMEDIATE` | P1 |
| POST | `/instances/{vulInfoID}/archive` | `INSTANCE_ARCHIVE` | P1 |
| POST | `/instances/{vulInfoID}/verify-fix` | `INSTANCE_VERIFY_FIX` | P1 |

### 4.4 向下游注入请求头（G6）

| Header | 来源 | 说明 |
|--------|------|------|
| `X-Partner-Id` | Token 上下文 | open-api-service **唯一**可信 Partner 标识 |
| `X-Request-Id` | 网关生成或透传 | 全链路追踪 |
| `Authorization` | 透传或剥离 | 建议剥离，避免业务层误用 Token 解析 partnerId |

### 4.5 白名单配置（🆕）

| 路径 | 是否免 Partner 鉴权 | 说明 |
|------|---------------------|------|
| `/oauth/token` | ✓ | Partner 取 Token |
| `/api/open/v1/oauth/token` | ✓ | 同上（若采用统一前缀） |
| `/api/open/v1/**` | ✗ | 必须 Partner Token |
| 门户路径 | — | 本网关**不存在**门户路径 |

### 4.6 基础设施复用

| 能力 | 说明 |
|------|------|
| Redis | 与 open-api-service **共用**集群；Partner 键前缀 `partner:token:*` 由 open-api-service 写入 |
| Sentinel | 🆕 按 `partnerId` 限流（可选） |
| Nacos | 独立 `partner-gateway` 服务注册与路由配置 |

## 5. open-api-service · 开放平台（🆕 · 身份 + 业务 + 治理）

**职责**：Partner 注册与凭证、Token 签发 / introspect、实现 `/api/open/v1/*`、**API 调用全生命周期记录与治理**、内部管理 API、数据隔离、幂等、Webhook 出站、SVMP 适配。  
**不负责**：门户 IAM、部侧加解密（属 Regulatory 模块）、Partner 公网 TLS / 连接层限流（属 partner-gateway）。

**服务内上下文**：从请求头 `X-Partner-Id` 读取（经 partner-gateway 注入）；**禁止**使用 body/query 中的 partnerId 作鉴权。

> **不必**为每个 REST 接口各写一套孤立实现。详见 [开放平台API治理与调用生命周期](./开放平台API治理与调用生命周期.md)。

### 5.0 双平面（执行 + 治理）

| 平面 | 模块 | P0 交付 |
|------|------|---------|
| **执行平面** | `handler/*`、`SvmpEngineAdapter`、业务表 | `POST/GET /tasks`、SVMP 对接、partner 隔离 |
| **治理平面** | `InvocationPipeline`、`ApiCatalog`、`InvocationService` | `api_operation` 种子、`api_invocation` 每次调用、requestId 全链路 |

```text
Partner HTTP → PartnerContextFilter → InvocationPipeline.start
              → TaskHandler / InstanceHandler（执行）
              → InvocationPipeline.finish → ApiResponse
```

| 项 | 说明 |
|----|------|
| capability | **授权标签**（网关 40301），不是 9 个独立微服务 |
| OpenAPI schema | **契约必须定义**；实现走 Handler + Pipeline |
| 监控 | **不对 Partner 暴露**；internal admin 查询 invocation（P1） |

### 5.1 Token 与鉴权接口（🆕）

| 方法 | 路径 | 调用方 | 说明 | 阶段 |
|------|------|--------|------|------|
| POST | `/oauth/token` | Partner（公网，经 partner-gateway） | 标准 OAuth2 `client_credentials` | P0 |
| POST | `/internal/partners/token` | Partner（备选 / 内网） | 简化 JSON 请求体 | P0 |
| POST | `/internal/token/introspect` | **partner-gateway** | 入参 token，返回 Partner 上下文 JSON | P0 |
| POST | `/internal/token/revoke` | 管理后台 | 吊销单个 token | P2 |

**Token 响应字段（对外）**

| 字段 | 说明 |
|------|------|
| `accessToken` / `access_token` | Bearer Token |
| `tokenType` | `Bearer` |
| `expiresIn` | 秒 |
| `partnerId` | 接入方 ID |

**Redis 缓存（open-api-service 写入）**

| 键 | 值 |
|----|-----|
| `partner:token:{sha256(accessToken)}` | `{ subjectType, partnerId, capabilities, clientId, expiresAt }` |
| `partner:credential:{clientId}` | 凭证元数据（不含明文 secret） |

> 与用户 Token 分离：用户 `ACCESS_TOKEN:*`（clover）；Partner `partner:token:*`（open-api-service）。`subjectType` 固定 `PARTNER`。

| POST | `/internal/events/publish` | SVMP / 编排器 | 内部事件，触发 Webhook 投递 |

### 5.6 Webhook 出站（open-api-service → Partner）

Partner **实现接收端**；平台主动 `POST` Partner 的 `callbackUrl`。

| 项 | 说明 |
|----|------|
| 触发模块 | open-api-service 事件投递器 |
| 签名头 | `X-Webhook-Signature`（HMAC-SHA256，密钥 `webhook_secret`） |
| 配置来源 | 本地 Partner 表（`partner_webhook_config`） |
| 成功响应 | HTTP 200，`{ "received": true }` |
| 失败 | 平台异步重试；业务码 `50002` |

**事件类型与 capability**

| eventType | 需 capability | 典型 payload 字段 | 阶段 |
|-----------|---------------|-------------------|------|
| `TASK_COMPLETED` | EVENT_SUBSCRIBE | taskId, extTaskId, status | P1 |
| `TASK_FAILED` | EVENT_SUBSCRIBE | taskId, reason | P1 |
| `INSTANCE_STATUS_CHANGED` | EVENT_SUBSCRIBE | vulInfoID, vulInfoStat | P1 |
| `INSTANCE_REMEDIATED` | EVENT_SUBSCRIBE | vulInfoID, vulInfoStat=5 | P1 |
| `INSTANCE_ARCHIVED` | EVENT_SUBSCRIBE | vulInfoID, vulInfoStat=9, lvRsn | P1 |
| `INSTANCE_VERIFY_FIX_COMPLETED` | EVENT_SUBSCRIBE | vulInfoID, vulInfoStat=6/7/10 | P1 |
| `EXPORT_READY` | EVENT_SUBSCRIBE | exportId, downloadUrl, exportStage | P2 |

`exportStage`：`TASK_COMPLETED` / `VERIFY_SCAN` / `VERIFY_FIX_SCAN`。

**职责**：扫描、漏洞处置、修复核验执行。  
**调用方**：仅 **open-api-service**（`SvmpEngineAdapter`）。Partner **不直连**。

### 6.1 引擎 REST 对照（平台内部）

| 开放平台动作 | SVMP 接口 | 入参要点 |
|--------------|-----------|----------|
| 创建任务 | `POST /task/create` | 扫描目标、模板；**不带** extTaskId |
| 创建前选设备 | `GET /device/list` | 按目标解析设备 |
| 查询进度 | `GET /task/progress` | **仅** SVMP taskId |
| 实例列表 | `POST /vuln/disposal/list` | SVMP taskId + 筛选 |
| 实例详情 | `GET /vuln/disposal/detail` | vulnDisposalId |
| 验证 / 修复 / 备案 | `POST /vuln/disposal/disposal` | vulnDisposalId + 处置参数 |
| 修复核验（触发引擎） | `POST /vuln/disposal/verify` | vulnDisposalId + 核验参数 |
| 修复核验（回写） | — | 平台状态机直接写 6/7/10，不调 SVMP |

## 7. Regulatory 模块 · 部侧合规（内部）

**职责**：工信部 2025 接口 3 入站、接口 4 回传；与开放平台 **并行**，不替代 `/api/open/v1`。  
**实现位置**：建议独立 `regulatory-adapter` 或 open-api-service 子模块（设计方案 §七）。

| `RegulatoryEgress.reportOrder` | 平台 → 部侧 | 接口 4 回传；`vulInfoLst` 组装 | P0（合规） |

**与开放平台关系**：Partner 触发的任务若需报送部侧，须关联 `orderID` 或进入同一编排上下文（设计方案 §7.6）。

## 8. 错误码职责划分

| code | 含义 | 产生组件 |
|------|------|----------|
| 40101 | 鉴权失败 | **partner-gateway**（Token / HMAC） |
| 40301 | 能力未开通 | **partner-gateway**（capability） |
| 42901 | 限流 | **partner-gateway** |
| 40001–40005 | 参数 / 状态机 / 资源 | open-api-service |
| 40003 | 跨 Partner 访问 | open-api-service（taskId / vulInfoID 不属于当前 partner） |
| 40901 | extTaskId 幂等冲突 | open-api-service |
| 50001 | 引擎调用失败 | open-api-service（SVMP 超时/异常） |
| 50002 | 回调投递失败 | open-api-service（Webhook） |

> Token 签发失败（错误 clientId/secret）由 open-api-service `POST /oauth/token` 返回 OAuth 标准错误或业务码，**不经过** partner-gateway 鉴权过滤器。
