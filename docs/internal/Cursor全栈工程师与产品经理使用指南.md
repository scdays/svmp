# Cursor 全栈工程师与产品经理使用指南

> 适用角色：同时承担产品规划、全栈研发、方案评审与交付验收的人。

本文沉淀 Cursor 在不同工作场景下的模式、功能与模型选择建议，目标是让需求分析、代码实现、调试、Review 与 PR 交付形成可复用流程。

---

## 1. 总体原则

一句话决策：

> 问代码用 Ask，想方案用 Plan，真改代码用 Agent，查 bug 用 Debug，视觉任务用 Gemini，复杂任务用 GPT-5.5 / Claude Opus，日常默认 Auto / Codex。

默认建议：

- **日常编码**：Auto 或 GPT-5.3 Codex。
- **复杂实现 / 长链路任务**：GPT-5.5。
- **架构设计 / 大型重构 / 强自治任务**：Claude Opus。
- **UI、截图、设计稿还原**：Gemini Pro。
- **全仓大上下文分析**：按需开启 Max Mode。
- **并行长任务或需要远端 PR**：Cloud Agents。

## 2. 场景与 Cursor 功能选择

| 场景 | 推荐功能 / 模式 | 使用方式 |
|------|------------------|----------|
| 理解代码、梳理架构、解释模块关系 | Ask Mode | 只读提问，不改文件。适合先摸清现状。 |
| 写功能、修 bug、补测试、局部重构 | Agent Mode | 让 Agent 搜索代码、编辑文件、运行命令并修复失败。 |
| 需求复杂、影响范围不确定 | Plan Mode | 先生成方案、风险点、任务拆分，确认后再执行。 |
| 报错、测试失败、难复现问题 | Debug Mode | 基于日志、栈追踪、复现步骤定位根因。 |
| 小范围局部改动 | Inline Edit / Cmd+K | 选中代码后直接描述修改目标。 |
| 日常补全、多行续写、自动补 import | Tab Completion | 编码时保持启用，用于快速连续编辑。 |
| 多任务并行、云端跑测试、自动开 PR | Cloud Agents | 适合 backlog、CI 修复、多仓库或长时间任务。 |
| PR 质量检查 | Bugbot / Agent Review | 关注逻辑 bug、安全风险、边界条件和测试缺失。 |
| 团队规范沉淀 | Rules / AGENTS.md | 固化代码风格、架构边界、测试要求和常见坑。 |

## 3. 模型选择表

| 任务类型 | 推荐模型 | 说明 |
|----------|----------|------|
| 普通问答、日常开发、成本敏感 | Auto | 自动平衡效果、速度与成本。 |
| 常规编码、修 bug、补测试 | GPT-5.3 Codex | 稳定默认值，适合大多数开发任务。 |
| 快速编辑、交互式小改 | Composer 2.5 | 适合快速工具调用和文件编辑。 |
| 多步骤复杂实现、长期任务 | GPT-5.5 | 推理和坚持性更强，适合复杂需求。 |
| 架构方案、大型重构、自治执行 | Claude Opus | 适合复杂代码库、计划与高质量实现。 |
| UI/UX、截图、设计稿、视觉差异 | Gemini Pro | 更适合图像 + 代码联合理解。 |
| 最复杂任务且不想手动选择 | Premium | 由 Cursor 选择强模型，成本更高。 |
| 超大上下文、全仓分析 | Max Mode | 只在确实需要时开启，避免无谓成本。 |

## 4. 推荐工作流

### 4.1 产品需求阶段

使用：

- Plan Mode
- GPT-5.5 / Claude Opus / Premium

提示词示例：

```text
请阅读当前代码库，分析实现这个需求需要改哪些模块，给出技术方案、影响范围、风险点、验收标准和待确认问题。先不要写代码。
```

产出应包含：

- 需求边界。
- 受影响模块。
- 技术方案。
- 风险与依赖。
- 验收标准。
- 可拆分任务。

### 4.2 开发实现阶段

使用：

- Agent Mode
- GPT-5.3 Codex；复杂时升级 GPT-5.5

提示词示例：

```text
根据刚才确认的方案实现该功能。请保持现有代码风格，补充必要测试，运行相关测试并修复失败。完成后给出变更摘要和验证结果。
```

执行要求：

- 先读代码和现有模式。
- 小步提交或保持清晰 diff。
- 有风险时补测试。
- 运行与改动相关的最小测试集。

### 4.3 Debug 阶段

使用：

- Debug Mode
- GPT-5.3 Codex / GPT-5.5

提示词示例：

```text
这是报错日志和复现步骤。请先分析最可能根因，定位相关代码，给出验证方法，再实施修复并运行测试。
```

注意事项：

- 不要只根据报错文本盲改。
- 优先复现或构造最小验证。
- 修复后补充能覆盖该问题的测试。

### 4.4 前端和 UI 阶段

使用：

- Agent Mode + 截图 / 设计稿
- Gemini Pro

提示词示例：

```text
请对比目标截图与当前页面，实现视觉调整。重点关注布局、间距、字体、颜色、响应式表现，并用浏览器验证最终效果。
```

适用任务：

- 设计稿还原。
- 视觉 bug。
- 响应式布局。
- 截图对比验收。

### 4.5 PR Review 阶段

使用：

- Bugbot
- Agent Review

提示词示例：

```text
请 review 当前 diff，重点检查逻辑 bug、边界条件、安全问题、性能风险和测试缺失。不要只关注格式问题。
```

Review 重点：

- 是否满足验收标准。
- 是否破坏兼容行为。
- 是否缺少异常路径处理。
- 是否缺少关键测试。
- 是否引入安全或权限风险。

## 6. 规则与团队沉淀

建议把重复出现的要求沉淀到项目规则：

- 代码风格和目录约定。
- 后端鉴权、租户隔离、权限校验要求。
- 前端组件、状态管理、接口调用约定。
- 测试要求。
- 禁止事项，例如绕过网关、硬编码密钥、跳过错误处理。

实践建议：

- 规则保持短小、明确、可执行。
- 不把通用格式规范复制进规则，交给 linter / formatter。
- 当 Agent 反复犯同类错误时再添加规则。
- 团队共享规则应提交到 Git。

## 7. 常用提示词模板

### 需求转方案

```text
你担任产品经理和技术负责人。请基于当前代码库分析该需求，输出实现方案、影响范围、风险点、验收标准和任务拆分。不要写代码。
```

### 方案转实现

```text
请按已确认方案实现。遵循现有代码风格，优先复用已有模块，补充必要测试，运行相关验证，并在最后总结改动和测试结果。
```

### 复杂问题排查

```text
请基于日志、复现步骤和相关代码定位根因。先列出假设和验证步骤，再修改代码。修复后运行测试证明问题已解决。
```

### PR 自查

```text
请 review 当前变更，优先找真实 bug、边界条件、安全风险、性能问题和测试缺口。按严重程度列出问题，并给出修改建议。
```

### UI 还原

```text
请根据目标截图调整当前页面，实现视觉一致。重点检查布局、间距、字体、颜色、状态和响应式表现，并输出验证截图或说明。
```
